LangGraph基础

LangGraph

什么是 LangGraph？

LangGraph 是一个低级别的编排框架和运行时，专门用于构建、管理和部署长期运行、有状态的智能体（agents）。

  • 低级别（Low-level）: 它不隐藏底层细节，让你对智能体的执行流程、状态管理等有完全的控制权。你需要自己定义图的结构、节点逻辑和状态。
  • 编排（Orchestration）: 它的核心任务是协调智能体内部各个组件（如大语言模型、工具、内存等）的执行顺序和交互方式。
  • 长期运行（Long-running）: 适用于需要长时间执行、可能等待外部输入或异步操作的智能体。
  • 有状态（Stateful）: 智能体能够记住过去的信息和交互历史，这对于复杂的、多步骤的任务至关重要。

LangGraph 的定位

  • 与 LangChain 的关系: LangGraph 是一个独立的框架，可以不依赖 LangChain 使用。但 LangChain 的高级智能体抽象（Agents）是建立在 LangGraph 之上的。如果你刚开始或需要更高层次的抽象，可以从 LangChain 的 Agents 开始；如果你需要更精细的控制和处理复杂工作流，则选择 LangGraph。
  • 专注点: LangGraph 专注于智能体编排所必需的核心能力，而不是像 LangChain 那样提供广泛的 LLM 组件集成。

LangGraph 的核心优势

  • 持久化执行 (Durable Execution): 即使遇到故障，也能从中断处恢复，支持长时间运行。
  • 人机交互 (Human-in-the-loop): 允许在执行过程中插入人工审核、干预或决策。
  • 全面的记忆 (Comprehensive Memory): 支持短期工作记忆（用于推理）和长期记忆（跨会话）。
  • 调试 (Debugging): 与 LangSmith 集成，提供强大的可视化和追踪功能。
  • 生产就绪部署 (Production-ready Deployment): 提供可扩展的基础设施来部署复杂的智能体系统。

🧠 总结

LangGraph 提供了一个强大而灵活的框架来构建复杂的、有状态的智能体。它的核心思想是将智能体的行为建模为一个有向图，其中节点是执行单元，边是执行流转，状态是贯穿整个执行过程的数据。通过这种方式，它可以优雅地处理需要记忆、规划、人机交互和错误恢复的复杂任务。

核心概念总览

LangGraph 的核心思想是将代理（Agent）的工作流程建模为一个有向图（Directed Graph）。这个图由三个关键组件构成：状态（State）、节点（Nodes）和边（Edges）。整个系统使用一种基于消息传递的算法（受 Google Pregel 启发）来执行，将程序执行划分为一系列离散的“超步”（Super-steps），使得复杂的、可循环的、并行的逻辑得以清晰表达。

Super-step（超级步骤）

在 LangGraph 中，“超级步骤”（Super-step）指的是图（Graph）执行中的一个单一循环或迭代。你可以把它想象成图计算模型（如 Pregel 模型）中的一个完整的时间步长。

  1. 定义：一个超级步骤代表了图中所有可以并行执行的节点完成其计算并产生输出的一次完整过程。它不是指单个节点的执行，而是指图中一个或多个节点执行完毕后，整个图的状态向前推进一步。
  2. 执行流程：
    • 计划（Planning）：在超级步骤开始时，LangGraph 会确定哪些节点需要在这个步骤中执行。这通常基于上一个超级步骤的输出和图的边（edges）。
    • 执行（Execution）：所有被计划的节点会并行（如果可能）或按顺序执行。
    • 状态更新（State Update）：所有节点执行完毕后，它们的输出会被聚合起来，用于更新图的全局状态。
    • 检查点保存（Checkpoint Saving）：在状态更新完成后，检查点器（Checkpointer）会自动将更新后的状态保存为一个新的检查点（Checkpoint）。这就是为什么文档中说“检查点在每个超级步骤后保存”的原因。
  3. 与检查点的关系：每次超级步骤结束，也就是图状态更新后，就会生成一个检查点。因此，一个图的执行历史就是由一系列按时间顺序排列的超级步骤检查点组成的。通过这些检查点，你可以精确地回溯到图执行过程中的任意一个中间状态。

  • 超级步骤 2: 执行 node_b。node_b 运行并返回 {"foo": "b", "bar": ["b"]}。这个返回值再次更新全局状态。由于没有下一个节点，图执行结束。最后一个检查点被创建，记录了最终状态。

所以，每一次图的状态发生改变并准备执行下一个（批）节点时，就完成了一个“超级步骤”，并随之保存了一个检查点。

Graph (图)

定义: 在 LangGraph 中，一个 Graph 是一个有向图，用于表示代理或应用程序的工作流程。它由节点（Nodes）和连接这些节点的边（Edges）组成。

作用: Graph 是整个工作流的蓝图。它定义了数据如何流动、计算在何处发生以及执行的顺序。通过将逻辑分解为节点和边，可以构建出复杂、动态且可维护的应用程序。

详细解释:

  • 工作流模型化: 它将传统的线性或分支代码结构转换为图形结构，这使得处理循环（如反思和重试）、并行任务（如同时调用多个工具）和复杂的决策树变得更加直观。
  • 状态管理: Graph 的核心在于其对共享状态（State）的管理。所有节点都可以读取和修改这个共享状态，从而实现信息在不同步骤间的传递。
  • 执行模型: 底层采用消息传递机制。当一个节点完成其计算后，它会将更新后的状态（或部分更新）沿着边发送给下一个或多个节点。接收节点被激活并开始执行。这种模式被称为“超步”，在每一步中，所有准备好执行的节点可以并行运行，然后进入下一步。
  • 生命周期: 图的执行始于虚拟的 START 节点，接收用户输入。当所有节点都变为非活动状态且没有待处理的消息时，图的执行终止于 END 节点。

State (状态)

定义: State 是一个共享的数据结构，代表了应用程序在某一时刻的快照。它是图中所有节点通信的中心枢纽。

作用: State 存储了工作流过程中需要的所有数据，确保了信息在不同节点间的持久性和一致性。

详细解释:

  • 共享内存: 所有节点都可以访问同一个 State 对象（或其快照）。当一个节点完成工作后，它会返回对 State 的更新，这些更新会被应用到全局 State 上，供后续节点使用。
  • 灵活性: State 可以是任何 Python 数据类型，但通常使用 TypedDict 或 Pydantic BaseModel 来定义一个结构化的 Schema，以便于类型检查和数据验证。
  • Reducer 函数: State 的每个字段都有一个对应的 Reducer 函数，该函数定义了当节点返回对该字段的更新时，如何将新值与旧值合并。默认行为是覆盖（Overwrite），但对于列表等类型，通常希望追加（Append），这时就需要自定义 Reducer。
  • 消息处理: 对于涉及对话历史的应用，State 中通常会有一个专门存储 Message 对象列表的字段。使用预设的 add_messages Reducer 可以智能地追加新消息并更新现有消息（如果它们具有相同的 ID）。

Nodes (节点)

定义: Node 是一个 Python 函数（同步或异步），它封装了图中的一部分具体逻辑。它接收当前的 State 作为输入，执行计算或产生副作用（如调用 LLM 或数据库查询），并返回对 State 的更新。

作用: Node 是执行实际工作的单元。它们负责处理数据、做出决策、调用外部服务，并将结果写回共享状态。

详细解释:

  • 函数式: Node 本质上是一个函数。这使得它们易于编写、测试和理解。LangGraph 内部会将其包装成 RunnableLambda，赋予其批处理、异步和追踪能力。
  • 输入: Node 总是接收当前的 state。此外，还可以选择性地接收 config（配置信息，如 thread_id、标签等）和 runtime（运行时上下文，如用户ID、数据库连接等）。
  • 输出: Node 返回的是对 State 的一个更新，而不是完整的 State。这个更新是一个字典，其键对应 State Schema 中的字段名。这些更新会被相应的 Reducer 处理后应用到全局 State 上。
  • 特殊节点: START 和 END 是两个虚拟节点，分别代表图的入口和出口。
  • 缓存: 为了优化性能，特别是对于计算成本高昂的节点，LangGraph 支持节点级缓存。可以通过 cache_policy 参数为节点配置缓存策略。

Edges (边)

定义: Edge 定义了节点之间的执行顺序和路由逻辑。它决定了一个节点执行完毕后，控制权应该交给哪个（或哪些）下一个节点。

作用: Edge 控制着图的执行流程，是实现条件分支、循环和并行的关键。

详细解释:

  • 固定边 (Normal Edges): 最简单的边，表示一个固定的、无条件的跳转。使用 add_edge(from_node, to_node) 添加。
  • 条件边 (Conditional Edges): 更强大的边，它依赖于一个路由函数（Routing Function）的返回值来决定下一步的去向。使用 add_conditional_edges(node, routing_function) 添加。路由函数接收当前状态，返回一个节点名称（字符串）或节点名称列表。
  • 入口点 (Entry Point): 通过 add_edge(START, node_name) 或 add_conditional_edges(START, routing_func) 来定义图启动时首先执行的节点。
  • 动态边 (Dynamic Edges): 通过从节点或工具返回 Send 或 Command 对象，可以在运行时动态决定下一个要执行的节点，这突破了静态图结构的限制。

作用: StateGraph 提供了一个接口，让你可以方便地定义状态 Schema、添加节点、配置边、设置入口点和终点，最后编译成一个可执行的图对象。

Compiling (编译)

定义: 编译是将通过 StateGraph 构建的逻辑结构（节点、边、状态）转换为一个可执行的、经过校验的图对象的过程。

作用: 编译是图从“设计”阶段进入“执行”阶段的关键步骤。它确保图的结构是有效的，并完成必要的初始化。

详细解释:

  • 结构校验: 编译器会检查图是否有效，例如是否存在孤立的节点（没有入边也没有出边）、所有引用的节点名是否都已定义等。
  • 运行时配置: 在编译时，可以传入一些运行时参数，如 checkpointer（用于持久化状态和恢复执行）、breakpoints（用于调试）和 cache（用于节点缓存）。
  • 必要步骤: 文档明确指出，必须先编译图，然后才能调用它。未编译的图对象无法执行。

Schema (状态模式)

定义: Schema 是对 Graph State 结构的正式描述，它定义了 State 对象中有哪些字段以及每个字段的类型。通常使用 TypedDict 或 Pydantic BaseModel 来定义。

作用: Schema 为 State 提供了类型安全和结构约束，使开发者能够清晰地知道状态中包含哪些数据。

详细解释:

  • 类型提示: TypedDict 提供了良好的 IDE 自动补全和静态类型检查支持，而 Pydantic BaseModel 则提供了更强大的数据验证功能。
  • 输入/输出分离: 可以定义独立的 input_schema 和 output_schema，使得图的公开接口与内部使用的完整 internal_state 相分离。这对于构建模块化和API友好的图非常有用。
  • 私有状态: 节点可以写入到不在 input_schema 或 output_schema 中但在 internal_state 中定义的“私有”通道，用于内部节点间通信。

Reducers (归约器)

定义: Reducer 是一个函数，它定义了当节点返回对 State 中某个特定字段的更新时，应该如何将这个新值与该字段的当前值进行合并。

作用: Reducer 提供了对 State 变更的精细控制，尤其是在处理列表、集合或复杂对象时。

详细解释:

  • 默认行为: 默认的 Reducer 行为是“覆盖”。如果节点返回 {"list_field": [1, 2]}，那么 State 中的 list_field 将被完全替换为 [1, 2]。
  • 自定义行为: 通过 typing.Annotated，可以为字段指定自定义的 Reducer。最常用的自定义 Reducer 是 operator.add，它用于列表拼接。
  • add_messages: 这是一个专门为 Message 对象列表设计的高级 Reducer。它不仅能拼接列表，还能识别消息 ID 并更新已存在的消息，非常适合对话场景。
  • Overwrite: 在某些情况下，即使字段指定了其他 Reducer，也可能需要强制覆盖。Overwrite 类型允许这样做。

START Node (起始节点)

定义: START 是一个特殊的、虚拟的节点，代表图执行的入口点。

作用: 它不执行任何逻辑，而是作为一个占位符，用于通过 add_edge 或 add_conditional_edges 来指定图开始执行时应调用的第一个节点。

详细解释:

  • 路由起点: 所有通往 START 的边都是无效的。你只能从 START 画出边到你的第一个真实节点。

Node Caching (节点缓存)

定义: Node Caching 是一种性能优化机制，它将节点的输出基于其输入进行缓存，当再次遇到相同输入时，直接返回缓存的结果，而无需重新执行节点的逻辑。

作用: 显著提高性能，特别是对于那些计算成本高、耗时长的节点（如调用昂贵的 LLM API）。

详细解释:

  • 键生成: 缓存键通常是基于节点输入的哈希值。可以通过 cache_policy.key_func 自定义键的生成方式。
  • TTL (Time To Live): 缓存项可以设置生存时间（以秒为单位）。过期后，缓存项会被清除，下次请求时会重新计算。如果未设置 TTL，则缓存项永不过期。
  • 配置: 缓存是在图编译时通过 cache 参数配置的。节点级别的缓存策略通过 add_node 的 cache_policy 参数设置。

Conditional Edges (条件边)

定义: Conditional Edges 是一种动态的边，它不直接指定下一个节点，而是通过调用一个路由函数（Routing Function）来决定下一个或几个要执行的节点。

作用: 实现复杂的条件逻辑、分支和决策树，使图的行为可以根据运行时状态动态改变。

详细解释:

  • 路由函数: 这是一个 Python 函数，接收当前的 state 作为唯一参数（或加上 config 和 runtime）。它必须返回一个字符串（目标节点名）或字符串列表（多个目标节点名，它们将并行执行）。
  • 并行执行: 如果路由函数返回一个节点名列表，那么这些节点将在下一个超步（Super-step）中并行执行。
  • 入口点: START 节点也可以拥有条件边，用于根据输入动态决定从哪个节点开始执行。

Send

定义: Send 是一个特殊的对象，可以从条件边的路由函数或节点函数中返回。它允许在运行时动态地指定下一个要执行的节点以及传递给该节点的具体状态。

作用: 实现高级的控制流模式，如 Map-Reduce。它允许在一个节点产生多个子任务（例如，为列表中的每个项目生成一个任务），并在不知道确切数量的情况下将它们分派给下游节点处理。

详细解释:

  • 动态分派: Send 克服了图结构在编译时必须固定的限制，允许在运行时创建新的执行路径。
  • 独立状态: 每个 Send 对象可以携带一个独立的状态副本，传递给目标节点。这与共享状态不同，它为每个分派的任务提供了隔离的上下文。
  • Map-Reduce 示例: 一个节点生成一个项目列表 [item1, item2, item3]，它可以返回 [Send("processor", {"item": item1}), Send("processor", {"item": item2}), ...]，从而动态地为每个项目创建一个到 processor 节点的调用。

Command

什么是持久化（Persistence）？

Durable execution

什么是持久化执行（Durable Execution）？

持久化执行是一种技术，它让一个工作流（Workflow）能够在关键节点保存进度，从而允许它暂停并在之后精确地从停止的地方恢复。这种机制特别适用于以下场景：

  • 人机交互（Human-in-the-loop）：用户可以在工作流运行中检查、验证或修改流程，然后继续。
  • 长时间运行的任务：这类任务可能因中断或错误而失败（例如，调用大语言模型超时）。
  • 断点恢复：通过保存已完成的工作，持久化执行确保工作流可以从上次记录的状态恢复，而无需重新处理之前的步骤 —— 即使延迟一周也可以。

LangGraph 的持久化层

LangGraph 内置的持久化层（Persistence Layer）为工作流提供了持久化执行能力，它确保每个执行步骤的状态都被保存到一个持久化存储（Durable Store）中。

  • 保证恢复：如果工作流被中断（无论是系统故障还是为了人机交互），它可以从中断处恢复。
  • 自动启用：如果你在 LangGraph 中使用了检查点器（Checkpointer），那么持久化执行就已经默认启用了。

持久化模式（Durability Modes）

LangGraph 支持三种持久化模式，允许你根据应用需求平衡性能和数据一致性：

模式 描述 优点 缺点
"exit" 仅在工作流退出（成功、失败或中断）时保存。 性能最佳 无法从执行中途的崩溃中恢复。
"async" 在下一步执行时异步保存更改。 性能良好，持久性较好 进程崩溃时有小概率丢失检查点。
"sync" 在下一步开始前同步保存更改。 持久性最高 性能开销较大。

🧱 实现持久化执行的要求

要利用 LangGraph 的持久化执行，你需要：

  1. 启用持久化：在工作流中指定一个检查点器（Checkpointer），它将保存工作流进度。
  2. 指定线程标识符（Thread ID）：在执行工作流时指定一个线程 ID，用于跟踪特定工作流实例的执行历史。
  3. 封装副作用和非确定性操作：将任何非确定性操作（如随机数生成）或带有副作用的操作（如文件写入、API 调用）包装在 task 中，以确保在恢复时不会重复执行这些操作。

⚠️ 确定性与一致重播（Determinism and Consistent Replay）

这是一个关键概念：

  • 不是从代码行恢复：当你恢复一个工作流运行时，代码不会从停止的同一行代码恢复。相反，它会找到一个合适的起点，从该起点继续执行。
  • 重播过程：这意味着工作流将从起点开始重播所有步骤，直到到达它停止的位置。

因此，在编写用于持久化执行的工作流时，必须注意：

  • 避免重复工作：如果一个节点包含多个带副作用的操作（如日志记录、文件写入），请将每个操作包装在单独的 task 中。这样，在恢复时，这些操作不会重复执行，其结果会从持久化层检索。
  • 封装非确定性操作：将任何可能产生非确定性结果的代码（如随机数生成）包装在 task 或 node 中。
  • 使用幂等操作：尽可能确保副作用（如 API 调用、文件写入）是幂等的。这意味着如果在工作流失败后重试该操作，它的效果与第一次执行相同。

Stream

概念解读：LangGraph 流式传输 (Streaming)

LangGraph 的流式传输系统是一种核心功能，它允许在图（Graph）执行过程中实时地、逐步地获取数据更新。这对于构建基于大型语言模型（LLM）的应用至关重要，因为它可以显著提升用户体验（UX）。当用户向 LLM 发起请求时，流式传输使得应用可以在 LLM 生成完整响应的过程中，就将已经产生的部分结果（如文本片段、状态变化等）即时展示给用户，从而减少了感知上的延迟。

核心优势：

  • 提升用户体验 (UX): 通过即时显示部分输出，避免了用户面对空白界面或长时间等待的焦虑感。
  • 实时反馈: 用户可以看到内容生成的过程，增加了互动性和透明度。
  • 灵活性: 支持多种类型的数据流，包括最终结果、中间状态、LLM 令牌（tokens）、调试信息和开发者自定义数据。

支持的流式传输模式 (Supported Stream Modes)

LangGraph 提供了多种预定义的流式传输模式，开发者可以通过在 stream 或 astream 方法中设置 stream_mode 参数来选择一种或多种模式。这使得开发者可以根据具体需求精确控制接收到的数据类型。

模式 (Mode) 描述 用途
values 在图的每一步执行后，流式传输整个状态（State）的完整值。 当你需要了解图执行过程中的每一个完整状态快照时使用。例如，你想看到 {"topic": "ice cream", "joke": ""} 如何变为 {"topic": "ice cream and cats", "joke": ""} 再变为 {"topic": "ice cream and cats", "joke": "This is a joke..."}。
updates 在图的每一步执行后，流式传输状态的更新部分（增量）。 当你只关心节点执行所带来的具体变化时使用。例如，一个节点返回 {"joke": "This is a joke..."}，那么 updates 模式只会流式传输这部分更新，而不是整个状态。
messages 从图中任何调用 LLM 的地方，流式传输 LLM 的令牌（token）或消息块（message chunk），以及相关的元数据。 用于实现 LLM 输出的实时逐字显示（打字机效果），并可以获取关于该次 LLM 调用的详细信息（如来自哪个节点）。
custom 从图内的节点或工具函数中流式传输开发者自定义的任意数据。 允许开发者发送进度条更新、日志、临时结果或其他任何自定义信息到流中。
debug 在图执行期间流式传输尽可能多的信息。 主要用于开发和调试，可以看到每个节点执行前后的状态、节点名称等详细内部信息。

如何使用：

  • 单一模式: stream_mode="values"
  • 多种模式: 将模式放入列表，如 stream_mode=["values", "messages", "custom"]。此时，流式迭代器会返回 (mode, chunk) 形式的元组，其中 mode 是字符串（如 "values"），chunk 是对应模式下的数据。

流式传输图状态 (Stream Graph State)

这是流式传输最基础也最重要的用途之一。通过 values 和 updates 模式，你可以实时监控图的状态是如何一步步演变的。

  • 按节点过滤: 检查 metadata["langgraph_node"] 是否等于你关心的节点名称。
  • 按标签过滤: 在初始化 LLM 时为其添加 tags，然后在循环中检查 metadata.get("tags")。

流式传输自定义数据 (Stream Custom Data)

这是最灵活的功能。你可以让节点或工具函数发送任意的自定义数据到流中，用于报告进度、传递中间结果或发送信号。

这个函数是获取一个“写入器”的入口。拿到写入器后，你可以随时向流中写入自定义数据。

与任何 LLM 集成 (Use with Any LLM)

custom 模式不仅限于发送简单的进度信息。它是一个强大的钩子，允许你将任何外部的、非 LangChain 标准的流式数据源（如直接调用 OpenAI API、调用其他公司的模型 API 等）整合进 LangGraph 的流式传输体系中。

这种机制极大地增强了 LangGraph 的通用性和扩展性。

Interrupts

LangGraph 的 中断（Interrupts） 是一种强大的机制，它允许你在图执行过程中的特定点暂停执行流，并等待来自外部的输入（例如人工干预、API 调用结果或来自另一个系统的信号）后，再继续执行。这使得构建需要人类参与决策、审批、审查或验证的 AI 应用（即“人机协作（Human-in-the-Loop）”模式）成为可能。

核心定义与作用

  1. 定义：
    • 中断是一个在节点内部调用的函数，它会立即停止当前节点及其下游的进一步执行。
    • 它将图的当前状态通过配置的检查点（Checkpointer）进行持久化保存。
    • 它返回一个指定的值（称为有效载荷，payload），该值会被传递给调用者，并放在返回结果的 __interrupt__ 字段中，用于向调用者说明图在何处以及为何被暂停。
  2. 作用：
    • 实现人机协作：最核心的作用是实现 Human-in-the-Loop。例如，在执行一个重要的数据库操作前，暂停并询问用户是否批准。
    • 动态暂停点：与静态的调试断点不同，interrupt() 可以放置在任何代码位置，并且可以根据运行时逻辑有条件地触发，使其非常灵活。
    • 状态保持：当图暂停时，其完整的内部状态（包括所有节点的状态）都会被保存。这意味着无论在何种状态下被中断，都能准确无误地从中断点恢复执行，即使是在出错状态。
    • 外部驱动：允许外部系统（如 UI、另一个服务）驱动图的执行流程，决定何时以及如何继续。

关键组成部分

  1. interrupt() 函数：
    • 位置：在你希望暂停图执行的节点函数内部调用。
    • 参数：接受任意 JSON-可序列化 的值（字符串、数字、列表、字典等）。这个值就是有效载荷，它会被存储并在图暂停时返回给调用者，通常用来描述需要外部做什么。
    • 行为：调用后，图的执行被挂起，当前状态被保存到检查点，有效载荷被返回。
  2. 检查点（Checkpointer）：
    • 作用：这是实现中断功能的基础设施。它负责在中断发生时持久化图的状态。
    • 重要性：在生产环境中，必须使用一个持久化的检查点（如基于数据库的 SqliteSaver, PostgresSaver），而不是内存中的检查点（MemorySaver），以确保状态不会因程序重启而丢失。
  3. **线程 ID (thread_id)**：
    • 作用：这是一个关键的配置项，它充当了图执行状态的唯一标识符和“指针”。它告诉检查点系统应该加载和恢复哪个特定的图执行实例。
    • 使用：在调用 graph.invoke(...) 时，必须通过 config={"configurable": {"thread_id": "unique_thread_id"}} 来指定。无论是首次启动还是后续恢复，都必须使用相同的 thread_id，否则无法正确恢复状态。
  4. **Command(resume=...)**：
    • 作用：这是恢复被中断的图执行的关键方法。
    • 参数：resume 参数接收一个 JSON-可序列化的值。这个值会作为 interrupt() 函数的返回值，传递回调用它的节点。
    • 行为：当再次调用 graph.invoke(Command(resume=...), config=...) 时，图会从最初调用 interrupt() 的那个节点的开头重新开始执行（这是关键点，详见“规则”部分），并将 resume 的值赋给 interrupt() 的返回变量。

重要规则与注意事项

  1. 不要将 interrupt 包装在通用 try/except 中：
    • interrupt 通过抛出一个特殊的异常来暂停执行。如果用一个通用的 try...except Exception: 捕获它，这个暂停信号就会被吞掉，图不会真正中断。应该只捕获特定的、你确实想处理的异常。
  2. 不要改变节点内 interrupt 的调用顺序：
    • 当节点恢复执行时，它会从头开始运行。LangGraph 会按顺序匹配之前中断的值和恢复的值。如果节点内的中断调用顺序改变了（例如因为条件判断），会导致值错配，引发错误。确保中断调用是确定性的、顺序一致的。
  3. 传递给 interrupt 的值必须是 JSON-可序列化的：
    • 为了兼容各种检查点后端，传递给 interrupt 的参数（有效载荷）只能是简单的数据类型（字符串、数字、布尔值、列表、字典），不能是函数、类实例等复杂对象。
  4. 中断前的副作用必须是幂等的（Idempotent）：
    • 幂等性：一个操作执行一次和执行多次，其最终结果是一样的。
    • 原因：由于节点恢复时会从头开始执行，中断前的所有代码（包括可能的副作用，如 API 调用、数据库写入）都会被重复执行。如果这些操作不是幂等的（例如，每次都创建一条新记录），就会导致不一致或错误。
    • 最佳实践：将非幂等的副作用（Side Effects）放在 interrupt 调用之后，或者放在单独的节点中执行。
  5. 子图中的中断：
    • 如果在父图的一个节点中调用了包含中断的子图，那么当子图发生中断时，父图和子图都会暂停。恢复时，父图会从调用子图的那个节点的开头重新开始，子图也会从触发中断的节点的开头重新开始。
  6. 静态中断（用于调试）：
    • interrupt_before 和 interrupt_after 是在编译或调用时设置的固定断点，主要用于开发和调试，不适合用于生产环境的人机交互逻辑。它们会强制在指定节点之前或之后暂停。

通过理解和遵循这些概念、定义、作用、代码实现和规则，你可以有效地在 LangGraph 中利用中断功能来构建强大、可控且支持人机协作的 AI 应用。

Time Travel

  • 定义: 在 LangGraph 中，“时间旅行”是一种强大的调试和探索功能。它允许开发者将一个图（Graph）的执行状态恢复到历史上的某个特定检查点（Checkpoint），从而可以重新运行或修改后续的执行流程。
  • 作用:
    • 理解推理过程 (Understand reasoning): 当一个复杂的、由模型驱动的代理（如 LLM 驱动的智能体）产生了预期的结果时，开发者可以回到产生该结果之前的步骤，逐步分析其决策链路，了解它是如何一步步得出结论的。
    • 调试错误 (Debug mistakes): 当代理出现错误或产生不理想的结果时，开发者可以回溯到错误发生之前的检查点，检查当时的上下文（Context）、变量和状态，精确定位问题发生的环节和原因。
    • 探索替代方案 (Explore alternatives): 在某个关键决策点之后，开发者可以修改当时的状态（例如，改变输入参数或中间结果），然后从该点继续执行，以此来测试不同的路径或假设，看是否能产生更优的结果。这在优化策略或测试不同场景时非常有用。
  • 核心机制: 时间旅行的核心在于 Checkpointer (检查点存储器)。Checkpointer 会持续记录图在每次节点执行后的状态快照（State Snapshot）。这些快照构成了执行历史。时间旅行就是选择一个历史快照，然后基于它开启一条新的执行分支。

关键组件部分

1.Checkpoint (检查点)

  • 定义: 检查点是图在特定时刻的完整状态快照。它包含了当时所有节点的输出、全局状态（State）的内容以及可能的其他元数据。检查点由 Checkpointer 存储，并通过唯一的 checkpoint_id 进行标识。
  • 作用:
    • 故障恢复: 如果图的执行因意外而中断，可以从最近的检查点恢复，而不是从头开始。
    • 时间旅行的基础: 它是实现时间旅行功能的数据来源。没有检查点，就无法将图的状态恢复到过去。
  • 注意事项: 检查点的创建频率和数量取决于 Checkpointer 的配置。频繁的检查点会占用更多存储空间，但能提供更细粒度的恢复能力。

2.Checkpointer (检查点存储器)

  • 定义: Checkpointer 是 LangGraph 中负责管理检查点的对象。它定义了如何保存、检索和更新图的状态快照。LangGraph 提供了多种内置的 Checkpointer，如 MemorySaver (用于内存中临时存储)、PostgresSaver (用于持久化存储到 PostgreSQL 数据库) 等。
  • 作用:
    • 状态持久化: 将图的执行状态保存到一个持久化或临时的存储后端。
    • 状态读取: 允许图从特定的 thread_id 和 checkpoint_id 恢复其状态。
    • 支持并发: 使图能够支持长时间运行和并发访问，因为状态被妥善管理。
  • 注意事项: 在生产环境中，必须使用持久化的 Checkpointer (如 PostgresSaver) 来确保数据不会因程序重启而丢失。InMemorySaver 仅适用于本地开发和测试。

3.State (状态)

  • 定义: State 是贯穿整个图执行流程的共享数据结构。它通常是一个字典或 TypedDict，用于在各个节点之间传递和累积信息。例如，在一个对话机器人中，State 可能包含 messages 列表、user_info、session_data 等。
  • 作用:
    • 信息共享: 节点可以通过读取和写入 State 来相互通信和共享数据。
    • 状态追踪: 记录图在整个执行过程中的动态变化。
    • 时间旅行的关键: 时间旅行操作的就是这个 State 在历史时刻的快照。
  • 注意事项: State 的设计应该清晰明确，避免过于复杂导致难以维护。每个节点返回的更新字典会被合并到当前 State 中。

4.Thread (线程)

  • 定义: Thread (线程) 代表一次独立的图执行会话。每一次对图的 invoke 或 stream 调用，如果指定了一个新的 thread_id，就会在一个新的线程中进行。同一个 thread_id 下的所有执行历史都属于同一次会话。
  • 作用:
    • 会话隔离: 不同的用户交互或不同的任务实例可以通过不同的 thread_id 来隔离，互不影响。
    • 历史追踪: 每个线程都有其独立的执行历史（由 Checkpointer 维护），可以通过 thread_id 来查询和管理该会话的历史。
  • 注意事项: 在时间旅行中，必须指定正确的 thread_id，否则无法找到对应的历史记录。

5.Interrupt (中断)

  • 定义: 中断（Interrupt）是一种在图执行过程中主动暂停的机制。开发者可以在编译图时通过 .interrupt_before(node_name) 或 .interrupt_after(node_name) 来设置静态中断点，也可以在运行时动态决定是否中断。当执行到达中断点时，会暂停并等待外部指令（如人工干预或检查）。
  • 作用:
    • 开发调试: 在开发阶段，静态中断点可以帮助开发者在特定节点处停下来检查状态。
    • 人机交互: 在生产环境中，中断可以作为一个人工审核或决策点（Human-in-the-Loop）。
    • 辅助时间旅行: 中断发生时，会自动创建一个检查点。因此，设置中断是定位一个特定历史检查点的便捷方式，然后可以从此处进行时间旅行。
  • 注意事项: 本章主要讨论的是“时间旅行”，中断在此处的作用是帮助开发者快速定位到一个可供恢复的检查点。

注意事项

  1. Checkpointer 必需: 要使用 get_state_history, update_state 和 invoke (with config containing checkpoint_id) 等功能进行时间旅行，图在编译时必须传入一个 checkpointer 实例。否则，图不会记录任何历史状态。
  2. Thread ID 至关重要: 在调用 get_state_history 或 update_state 时，必须提供与原始执行相同的 thread_id，否则无法找到正确的会话历史。
  3. checkpoint_id 的作用: checkpoint_id 用于精确地定位到历史中的某一个特定状态。get_state_history 用于查找它，update_state 和 invoke (with config) 用于引用它。
  4. invoke(None, config) 的特殊性: 当你想从一个检查点恢复执行时，invoke 的第一个参数必须是 None。第一个参数是初始输入，从检查点恢复意味着不需要新的输入，而是从历史状态继续。
  5. update_state 的副作用: update_state 会创建一个新的检查点，并返回其新的 config。这使得你可以从一个修改过的状态分支出去，形成新的执行历史，而原始历史保持不变。这非常适合探索替代路径。
  6. get_state_history 的排序: 该方法返回的状态列表是按时间倒序排列的（最新的状态在列表开头）。在遍历时需要注意索引的含义。
  7. 生产环境考虑: 示例中使用了 InMemorySaver，它在程序重启后数据会丢失。在生产环境中，必须使用 PostgresSaver 或其他持久化 Checkpointer。
  8. 状态结构一致性: update_state 传入的 values 字典必须与定义的 State 结构兼容，否则可能导致图执行失败。

Memory

核心概念

1.Memory (记忆)

  • 定义: AI 应用程序（尤其是对话式代理）需要一种机制来在多次交互之间共享和保留上下文信息。这种机制就是“记忆”（Memory）。它使得代理能够记住之前的对话内容、用户的偏好、或者重要的事实，从而提供连贯且个性化的体验。
  • 作用:
    • 上下文连续性: 在多轮对话中，让代理“记得”之前说过的话和问过的问题。
    • 个性化: 存储用户特定的信息（如姓名、偏好），以便在未来的交互中使用。
    • 长期学习: 积累和利用跨会话的知识和经验。
  • 类型: LangGraph 中区分了两种主要的记忆类型：短时记忆（Short-term Memory）和长时记忆（Long-term Memory）。

2.Short-Term Memory (短时记忆)

  • 定义: 短时记忆，也称为线程级（thread-level）持久化，主要用于存储单次会话（由 thread_id 标识）内的上下文。它记录了当前对话的完整消息历史，使得代理能够在单次多轮对话中保持连贯性。它本质上是图（Graph）状态（State）在一次特定会话中的持续化。
  • 作用:
    • 支持多轮对话: 让代理能够看到并回应用户之前的消息。
    • 状态恢复: 在单次会话中，即使程序短暂中断，也可以从上次的检查点恢复状态继续对话。
  • 实现: 通过 LangGraph 的 Checkpointer 来实现。每次图执行完一个节点后，其状态（包括消息历史）会被保存到 Checkpointer 中。下次调用时，可以从 Checkpointer 中恢复该状态。
  • 生命周期: 与单个 thread_id 相关联，通常在会话结束后可以被清理。

3.Long-Term Memory (长时记忆)

  • 定义: 长时记忆用于存储跨越多个会话的用户特定或应用程序级别的数据。它可以存储用户档案、历史偏好、过往交互的摘要等信息，这些信息比单次会话的消息历史更为持久和结构化。
  • 作用:
    • 用户画像: 长期积累和存储关于特定用户的信息。
    • 跨会话知识: 在不同会话之间共享和检索重要的背景信息。
    • 语义搜索: 支持基于语义相似性的信息检索，而不仅仅是关键词匹配。
  • 实现: 通过 LangGraph 的 Store 来实现。开发者可以在节点内部主动地向 Store 中 put (存储) 数据，并在需要时 search (搜索) 或 aget (获取) 数据。
  • 生命周期: 通常与用户 ID 或应用范围相关联，生命周期远超单次会话，需要手动管理或定期清理。

4.Checkpointer (检查点存储器)

  • 定义: Checkpointer 是 LangGraph 中负责管理 短时记忆 的核心组件。它是一个对象，定义了如何将图的状态（State）序列化并持久化到存储后端（如内存、数据库），以及如何反序列化并恢复该状态。它是实现短时记忆和时间旅行（Time Travel）功能的基础。
  • 作用:
    • 状态持久化: 在图执行的每个步骤后，将当前的 State 保存起来。
    • 状态恢复: 根据 thread_id 和 checkpoint_id 从存储中加载 State，使图能够从中断处恢复执行。
    • 支持多会话: 通过 thread_id 区分不同的会话历史。
    • 启用高级功能: 为时间旅行、状态管理和调试提供数据支持。
  • 实现:
    • In-Memory: InMemorySaver，将状态存储在内存中，速度快，但程序关闭后数据会丢失，仅适用于开发和测试。
    • Database-Backed: 如 PostgresSaver, MongoDBSaver, RedisSaver 等，将状态存储在数据库中，数据持久，适用于生产环境。
  • 配置: 在编译图时，通过 graph.compile(checkpointer=...) 传入一个 Checkpointer 实例。

5.Store (存储)

  • 定义: Store 是 LangGraph 中负责管理 长时记忆 的组件。它是一个键值对（key-value）存储系统，允许开发者在节点内部进行更灵活的数据读写操作。与 Checkpointer 不同，Store 通常用于存储非图状态的、更结构化的数据。
  • 作用:
    • 长期数据存储: 存储用户偏好、历史记录、知识库片段等。
    • 数据检索: 支持按命名空间（namespace）和键（key）进行数据检索。
    • 语义搜索: 某些 Store 实现（如配置了嵌入模型的 InMemoryStore）支持基于语义相似性的搜索。
  • 实现:
    • In-Memory: InMemoryStore，同 Checkpointer，仅用于开发测试。
    • Database-Backed: 如 PostgresStore, RedisStore 等，用于生产环境。
  • 配置: 在编译图时，通过 graph.compile(store=...) 传入一个 Store 实例。

6.Runtime (运行时)

  • 定义: Runtime 是一个在节点函数执行期间注入的对象，它为节点提供了访问图的运行时环境的能力，特别是对 store 和 context 的访问。它遵循依赖注入模式，简化了对共享资源的访问。
  • 作用:
    • 访问 Store: 节点可以通过 runtime.store 来读写长时记忆。
    • 访问 Context: 节点可以通过 runtime.context 来获取在图调用时传入的上下文信息（如用户 ID），这对于在 Store 中使用正确的命名空间至关重要。
  • 使用: 在节点函数的签名中声明 runtime: Runtime[ContextType] 参数，LangGraph 会在调用时自动注入。

7.Context (上下文)

  • 定义: Context 是开发者定义的一个数据结构（通常是 @dataclass 或 TypedDict），用于在图的单次调用中传递一些元数据或配置信息。它不是图的 State，而是伴随整个调用过程的上下文。
  • 作用:
    • 传递元信息: 如 user_id, session_id, api_key 等，这些信息对于节点的逻辑很重要，但不属于图的主状态。
    • Store 命名空间: 通常用于在 Store 中构建命名空间，以隔离不同用户或场景的数据。
  • 使用: 定义好 Context 类型后，在 StateGraph 的构造函数中通过 context_schema=ContextClass 声明。在调用图时，通过 graph.invoke(..., context=ContextClass(...)) 传入具体实例。

8.Namespace (命名空间)

  • 定义: 在 LangGraph Store 中，Namespace 是一个用于组织和隔离存储数据的键。它通常是一个元组，例如 (user_id, "memories")，用于将不同用户或不同类型的数据分开存储。
  • 作用:
    • 数据隔离: 确保用户 A 的数据不会与用户 B 的数据混淆。
    • 有序组织: 将相关的数据归类到一起，方便管理和查询。

9.Semantic Search (语义搜索)

  • 定义: 一种搜索技术，它不仅基于关键词匹配，还基于内容的含义（语义）来查找相关信息。在 LangGraph Store 中，这意味着可以查询一个句子，并找到在意义上相似的存储项目，即使它们不包含完全相同的词语。
  • 作用:
    • 更智能的检索: 提高从长时记忆中找到相关信息的准确性。
    • 自然语言交互: 使代理能够以更自然的方式回忆起过去的信息。
  • 实现: 需要为 Store 配置一个嵌入模型（Embedding Model），该模型能够将文本转换为向量，然后在向量空间中进行相似度计算。

10.Managing Short-Term Memory (管理短时记忆)

  • 定义: 由于 LLM 的上下文窗口有限，随着对话轮次增加，消息历史可能会变得过长，超出模型的处理能力。管理短时记忆是指一系列策略，用于控制和修剪消息历史的长度，同时尽可能保留重要信息。
  • 作用:
    • 防止超限: 确保传递给 LLM 的消息不超过其最大上下文长度。
    • 优化性能: 减少不必要的计算开销。
    • 保持相关性: 保留最相关的历史信息。
  • 策略:
    • Trim Messages (裁剪消息): 根据 Token 数量或消息数量，保留最新的或最重要的消息，丢弃早期的消息。常用 langchain_core.messages.utils.trim_messages。
    • Delete Messages (删除消息): 使用 RemoveMessage 对象永久性地从图的 State 中移除特定消息。
    • Summarize Messages (总结消息): 使用 LLM 将早期的对话历史总结成一段简短的文字，并将其替换原始的长消息列表。
    • Manage Checkpoints (管理检查点): 通过 Checkpointer API 查看和删除历史检查点。

11.Checkpoint Management (检查点管理)

  • 定义: 涉及对 Checkpointer 中存储的检查点进行操作，如查看、遍历和删除。
  • 作用:
    • 调试: 查看特定时间点的图状态。
    • 监控: 了解图的执行历史。
    • 数据清理: 删除不再需要的旧会话数据，释放存储空间。
  • 操作:
    • View State: 使用 graph.get_state(config) 查看特定检查点的快照。
    • View History: 使用 graph.get_state_history(config) 获取整个会话的检查点历史。

5.管理短时记忆 - 裁剪消息 (Trim Messages)

trim_messages 是一个强大且灵活的工具，旨在帮助开发者安全、智能地裁剪聊天历史记录（即消息列表），以确保其符合大语言模型（LLM）的上下文长度限制（Context Window），同时还能维持对话的有效性和连贯性。

该函数接收一个消息列表和一系列配置参数，然后返回一个经过裁剪的新消息列表。裁剪的核心依据是 max_tokens 和 token_counter，它会根据指定的 strategy（例如保留最新的消息）来移除多余的、通常是较早的消息，直到剩余消息的总“Token”数满足要求。

参数详解:

messages (必需)

  • 类型: Iterable[MessageLikeRepresentation] | PromptValue
  • 说明: 这是要被裁剪的原始消息序列。通常，这是一个包含 HumanMessage, AIMessage, SystemMessage 等对象的列表。

max_tokens (必需)

  • 类型: int
  • 说明: 这是裁剪操作的硬性上限。函数会持续移除消息，直到剩余消息的总 Token 数 小于或等于 此值。

token_counter (必需)

  • 类型: Callable[[list[BaseMessage]], int] | Callable[[BaseMessage], int] | BaseLanguageModel | Literal["approximate"]
  • 说明: 这是一个至关重要的参数，用于定义如何计算一个或多个消息的“Token”数量。它有多种传入方式：
    • Literal["approximate"]: 字符串快捷方式，等同于传入 count_tokens_approximately 函数。这是一种快速、近似的计算方法，非常适合在高频调用（热路径）上使用，因为它速度快，但精度稍低。
    • Callable: 一个可调用函数。你可以传入一个自定义的函数，该函数接收一个消息列表并返回其 Token 总数。这为你提供了完全的控制权。
    • BaseLanguageModel: 一个语言模型实例（如 ChatOpenAI(...)）。函数内部会调用该模型的 get_num_tokens_from_messages 方法来进行精确计算。这是最精确但可能也是最慢的方式。
    • len: 当你传入 len 函数时，max_tokens 就不再是 Token 数，而是消息的数量。这会将裁剪逻辑从 Token 计数转换为简单的消息计数。

strategy (可选，默认: "last")

  • 类型: Literal["first", "last"]
  • 说明: 定义了裁剪消息的策略方向。
    • "last": （默认） 保留列表中最后面的几条消息，直到总 Token 数满足 max_tokens。这在对话场景中最为常用，因为它保留了最新的上下文。
    • "first": 保留列表中最前面的几条消息，直到总 Token 数满足 max_tokens。这会丢弃最新的对话，通常不符合对话历史的需求。

allow_partial (可选，默认: False)

  • 类型: bool

  • 说明: 当设置为 True

    时，如果在裁剪过程中，发现某条消息加入后会使总 Token 数超过 max_tokens，但移除这条消息又会使总 Token 数远低于 max_tokens，那么函数会尝试将这条单条消息的内容进行分割，只保留其中的一部分。

    • 如果 strategy='last'，则保留该消息的后半部分内容。
    • 如果 strategy='first'，则保留该消息的前半部分内容。
    • 如果为 False（默认），则整条消息要么被保留，要么被完全移除。

end_on (可选，默认: None)

  • 类型: str | type[BaseMessage] | Sequence[str | type[BaseMessage]] | None
  • 说明: 指定裁剪后消息列表的结尾消息类型。如果指定了此参数，函数会找到最后一次出现该类型消息的位置，并忽略该位置之后的所有其他消息。
    • 示例: end_on=("human", "tool") 表示裁剪后的列表最后一项必须是 HumanMessage 或 ToolMessage。这可以避免将 AI 的回复作为上下文的结尾，因为通常模型需要新的用户输入才能继续。

start_on (可选，默认: None)

  • 类型: str | type[BaseMessage] | Sequence[str | type[BaseMessage]] | None
  • 说明: 仅在 strategy='last' 时有效。指定裁剪后消息列表的起始消息类型。函数会找到第一次出现该类型消息的位置，并忽略该位置之前的所有其他消息（但不会影响因 include_system=True 而保留的位于索引 0 的 SystemMessage）。
    • 示例: start_on="human" 表示裁剪后的列表第一项必须是 HumanMessage。

include_system (可选，默认: False)

  • 类型: bool
  • 说明: 仅在 strategy='last' 时有效。如果原始消息列表的第一条是 SystemMessage，并且此参数设为 True，则这条 SystemMessage 会被强制保留，即使它超出了 max_tokens 的限制。这在需要始终保留系统指令给 AI 的场景中非常有用。

text_splitter (可选，默认: None)

  • 类型: Callable[[str], list[str]] | TextSplitter | None
  • 说明: 当 allow_partial=True 时，此参数才生效。它定义了如何将单条消息的内容进行拆分。你可以传入一个自定义的字符串分割函数，或者一个来自 langchain_text_splitters 模块的 TextSplitter 实例。如果未指定，函数会默认按换行符 \n 进行分割。

返回值:

  • 类型: list[BaseMessage]
  • 说明: 返回一个全新的、经过裁剪的消息列表。

异常:

  • ValueError: 当传入了两个互不兼容的参数，或者指定了一个无法识别的 strategy 时，会抛出此异常。

6.管理短时记忆 - 删除消息 (Delete Messages)

RemoveMessage 是一个类。它的设计目的非常明确：作为一个特殊的指令，用于标记并删除聊天历史中具有特定ID的其他消息。这在需要动态清理或编辑对话上下文的复杂Agent工作流中非常有用。

class RemoveMessage(BaseMessage)

  • 继承关系: RemoveMessage 继承自 BaseMessage 基类。这意味着它在结构上与 HumanMessage, AIMessage 等消息类型相似，可以被放入消息列表中，并被LangChain的工作流组件（如Graph）识别和处理。

属性详解:

type: Literal["remove"] = "remove"

  • 含义: 这是一个类属性，固定为字符串 "remove"。它是一个标识符，在序列化（如转为JSON）或在工作流中处理时，用于区分这个特殊的消息类型和其他常规的消息类型（如 "human", "ai"）。
  • 作用: 工作流引擎可以根据这个 type 来判断这是一个删除指令，而不是一个需要显示给AI模型的内容。

构造函数详解:

def __init__(self, id: str, **kwargs: Any)

这是创建一个 RemoveMessage 实例的方法。

  • id: str (必需)
    • 含义: 这是 RemoveMessage 的核心参数。它代表了待删除消息的唯一标识符。
    • 作用: 当工作流处理到这个 RemoveMessage 时，它会查找整个消息历史中 id 字段与此值相匹配的消息，并将其从历史记录中移除。因此，要使用 RemoveMessage，前提是你在创建原始消息时，需要为其分配一个唯一的ID（例如 HumanMessage("Hello", id="msg_123")）。
  • **kwargs: Any (可变关键字参数)
    • 含义: 允许在创建 RemoveMessage 时传入其他任意的字段。
    • 作用: 提供了扩展性，可以添加元数据或其他上下文信息。
  • 构造函数内部逻辑
    • if kwargs.pop("content", None): ...
      • 检查: 代码首先检查是否有人试图通过 kwargs 传入一个 content 字段（例如 RemoveMessage(id="xxx", content="some text")）。
      • 拒绝: 如果检测到 content，它会立即弹出（pop）这个键值对，并抛出一个 ValueError。这是因为 RemoveMessage 本身不存储任何要显示的内容，它只是一个删除指令，携带 content 是没有意义的，甚至可能是错误的。
    • super().__init__("", id=id, **kwargs)
      • 调用父类: 这行代码调用了父类 BaseMessage 的构造函数。
      • 空内容: 传入了一个空字符串 "" 作为 content。这再次强调了 RemoveMessage 不包含任何内容。
      • 传递ID: 将传入的 id 参数传递给父类，确保这个 RemoveMessage 自身也有一个ID（尽管它自己的ID与它要删除的那个消息的ID是两回事，但通常我们更关心它要删除的目标ID）。
      • 传递其他参数: 将 **kwargs 中剩余的其他参数也一并传递给父类。

注意事项

  1. thread_id 必须: 无论是使用 Checkpointer 还是 Store，只要涉及到会话管理，thread_id 都是 config 中必需的配置项，用于区分不同的执行流。
  2. 生产环境 Checkpointer: 绝对不要在生产环境中使用 InMemorySaver，因为它无法持久化数据。必须使用 PostgresSaver、MongoDBSaver 或 RedisSaver 等数据库后端的 Checkpointer。
  3. 生产环境 Store: 同样，生产环境中的长时记忆也必须使用数据库后端的 Store。
  4. setup() 方法: 大多数数据库后端的 Checkpointer 和 Store（如 PostgresSaver, RedisStore）在首次使用前都需要调用 setup() 方法来初始化数据库表或集合。
  5. context 的传递: 在使用长时记忆时，context (尤其是 user_id) 对于正确地在 Store 中划分命名空间至关重要。必须在调用 graph.invoke 时提供 context 对象。
  6. Runtime 的注入: 要在节点中访问 store 或 context，必须在节点函数签名中正确声明 runtime: Runtime[ContextType]。
  7. 删除消息的限制: 删除消息时要非常小心，确保删除后剩余的消息历史仍然符合 LLM 的要求（例如，不能以 Assistant 消息结尾而没有对应的 Tool 消息，或消息必须以 Human 开始等）。
  8. 状态管理: MessagesState 是一个特殊的 State，其 messages 键具有特殊的 add_messages reducer，这使得添加和删除消息的操作成为可能。自定义 State 需要正确配置 Reducer。
  9. 子图 (Subgraphs): 如果图包含子图，只需要在编译父图时提供 Checkpointer，它会自动传播到子图。
  10. 语义搜索: 启用 Store 的语义搜索功能需要额外配置一个嵌入模型 (init_embeddings)。
