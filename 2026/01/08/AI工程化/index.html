AI工程化

AI工程化的定义

将AI研究成果转化为稳定（可靠）、可维护（可控）、可扩展的生产系统。

架构

Agent AI 的 8 逻辑层架构（类比OSI模型）

一个真正的“AI Agent”不是单一模型，而是一个分层、协同、自治的复杂系统，类似于人类大脑 + 身体 + 社会组织的结合。

Layer 1: Infrastructure Layer（基础设施层）

功能：提供算力和数据存储基础

  • 关键组件：
    • GPU/TPU（NVIDIA A100/H100, Google TPU）
    • 云平台（AWS, Azure, GCP）
    • 数据库（MongoDB, PostgreSQL）
    • 存储（S3, Object Storage）
    • 网络（5G, Wi-Fi 6, CDN）
    • 工具链（Prometheus, Grafana）

✅ 作用：支撑所有上层计算与通信需求

💡 类比：人体的“骨骼、肌肉、神经系统”

Layer 2: Agent Internet Layer（智能体互联网层）

功能：实现智能体之间的互联互通

  • 核心协议：
    • **ACP (Agent-to-Agent Communication Protocol)**：智能体间通信标准
    • **TAP (Tool Abstraction Protocol)**：统一工具调用接口
    • **AAP (Agent API Protocol)**：对外暴露服务
    • **FAP (Function Call Protocol)**：函数调用规范
  • 支持能力：
    • 多智能体协作（Multi-Agent Systems）
    • 智能体身份管理（Agent Identity & State）
    • 工具共享与发现
    • 智能体网络（Agent Mesh Networks）

✅ 作用：让 AI 智能体像“联网设备”一样互相通信、合作、竞争

💡 类比：互联网 → 让电脑彼此连接；这里是“AI 世界的互联网”

Layer 3: Protocol Layer（协议层）

功能：定义智能体行为的规则与契约

  • 关键协议：
    • **A2A (Agent-to-Agent Protocol)**：智能体之间交互的通用语言
    • **ACP (Capability Protocol)**：能力注册与发现
    • **AGORA (Agent Governance & Resource Allocation)**：资源调度与权限控制
    • TAP / FAP / AAP：工具、函数、API 调用协议

✅ 作用：确保不同厂商、不同类型的智能体可以互操作

💡 类比：TCP/IP 是互联网的基础协议 → A2A 是 AI 智能体世界的“TCP/IP”

Layer 4: Tooling & Enrichment Layer（工具与增强层）

功能：为智能体提供“技能包”和“知识扩展”

  • 主要模块：
    • **External Augmented Generation (EAG)**：外部增强生成（如 RAG）
    • Vector DBs：向量数据库（FAISS, Pinecone）
    • Function Calling Tools：可调用的函数（如天气、支付、邮件）
    • Knowledge Bases：知识库（维基百科、企业文档）
    • Environment Interfaces：与现实世界交互的接口（机器人、IoT 设备）

✅ 作用：赋予智能体“动手能力”和“知识获取能力”

💡 类比：人类的手、脚、眼睛、耳朵 → 智能体的“感官”和“行动器官”

Layer 5: Cognition & Reasoning Layer（认知与推理层）

功能：智能体的“大脑”，负责思考和决策

  • 核心能力：

    • **Planning (PL)**：任务规划（如“先查订单 → 再联系客服”）
    • **Self-improvement (SI)**：自我优化（如试错学习）
    • **Reasoning Engine (RE)**：逻辑推理引擎
    • **Decision-Making (DM)**：决策制定（基于规则或概率）
    • **Feedback Loop (FL)**：反馈闭环（不断迭代改进）
    • Reactivity & Adaptation：对环境变化做出反应

✅ 作用：让智能体不仅能执行命令，还能自主思考、计划、适应

💡 类比：人类的大脑皮层 → 负责高级思维

Layer 6: Memory & Personalization Layer（记忆与个性化层）

功能：智能体的“记忆系统”和“个性塑造”

  • 关键组件：

    • **Working Memory (WM)**：短期记忆（当前对话上下文）
    • **Long-Term Memory (LTM)**：长期记忆（用户偏好、历史行为）
    • **Identity (ID)**：智能体身份（名字、角色、风格）
    • Personal Profile：个人档案（语气、专业领域）
    • Behavior Modeling：行为建模（如何说话、何时介入）

✅ 作用：让智能体具备“记忆”和“个性”，不再像机器人那样冷冰冰

💡 类比：人类的记忆 + 人格特质

Layer 7: Application Layer（应用层）

功能：面向用户的实际应用场景

  • 典型应用类型：

    • Functional Agents：功能型智能体（如客服、销售助手）
    • Enterprise Agents：企业级代理（如财务分析、HR 自动化）
    • Research Agents：科研智能体（自动写论文、做实验）
    • Platform Agents：平台级代理（如安全监控、合规审计）
    • Collaborative Agents：协作型智能体（多人会议助手）

✅ 作用：将底层能力转化为具体业务价值

💡 类比：手机上的 App → 用户直接使用的“软件”

Layer 8: Ops & Governance Layer（运营与治理层）

功能：确保智能体系统的安全性、合规性和可持续运行

  • 关键模块：

    • Deployment Pipelines：部署流水线（CI/CD for Agents）
    • No-Code/Low-Code Builders：低代码开发平台（非程序员也能创建 Agent）
    • Observability Tools：可观测性（日志、监控、追踪）
    • Governance & Policy Engines：治理策略引擎（防止越权、偏见）
    • Resource Management：资源调度（CPU、内存、预算）
    • Logging & Auditing：日志记录与审计（满足合规要求）
    • Trust Frameworks：信任机制（验证智能体身份、行为可信度）

✅ 作用：让 AI 智能体系统可控、可信、可管

🌐 整体架构总结

层级 核心职能 对应人类类比
1. 基础设施 算力与存储 骨骼、肌肉、神经
2. 智能体互联网 通信与连接 互联网
3. 协议层 交互规则 TCP/IP 协议
4. 工具与增强 技能与知识 手、脚、感官
5. 认知与推理 思考与决策 大脑皮层
6. 记忆与个性 记忆与人格 记忆 + 个性
7. 应用层 实际用途 手机 App
8. 运营与治理 安全与管理 IT 部门 + 法务

六层架构（实际开发）

1.Infrastructure Layer（基础设施层）

这是整个技术栈的地基，提供计算资源和存储能力。

  • 主要组件：
    • 云平台：Google Cloud TPU、AWS EC2 P4 / P5 Instances、Microsoft Azure NDv5 / Ncas Series、Oracle Cloud Infrastructure (OCI)、Lambda Labs GPU Cloud、Intel Habana Gaudi2
    • 硬件加速器：NVIDIA DGX Systems、Paperspace、CoreWeave、IBM Vela

✅ 作用：提供强大的算力支持，尤其是GPU/TPU等专用芯片，用于训练和运行大型模型。

💡 没有这层，就没有足够的计算能力支撑后续所有AI模型的训练与推理。

2.Architecture Layer（架构层）

这一层定义了模型的基本结构和学习方式。

  • 主流架构类型：
    • Transformer（如BERT、GPT系列）
    • Diffusion Models（扩散模型，常用于图像生成）
    • Autoencoders（自编码器）
    • CNN（卷积神经网络）
    • RNN（循环神经网络）
    • GANs（生成对抗网络）
    • Perceiver IO
    • Retrieval-Augmented Generation (RAG)
    • MoE（Mixture of Experts，专家混合）
    • SSM（State Space Models）

✅ 作用：决定模型如何处理数据、提取特征、生成内容。

💡 不同任务选择不同架构，例如：

  • 文本生成 → Transformer
  • 图像生成 → Diffusion + GANs
  • 长序列建模 → SSM 或 RNN

3.ML Foundation Model Layer（机器学习基础模型层）

这是“大模型”所在的一层，即我们常说的基础模型（Foundation Models）。

  • 代表模型：
    • GPT-4（OpenAI）
    • Claude（Anthropic）
    • LLaMA 3（Meta）
    • Mistral
    • Falcon（TII）
    • PaLM 2（Google）
    • Ouro（xAI）
    • Command R+（Cohere）
    • Yi（01.AI）
    • Gemini（Google）

✅ 作用：这些是经过海量数据预训练的通用模型，具备强大的泛化能力，可以微调或直接使用来完成各种下游任务。

💡 这些模型是生成式AI的核心“大脑”。

4.LLMLayer（大语言模型层）

这一层是基于基础模型构建的特定用途的大语言模型（LLMs），通常已优化或适配特定场景。

  • 例子：
    • Llama-Guard（安全过滤）
    • BlenderBot（Meta，对话系统）
    • Phi（Microsoft，小参数高效模型）
    • GPT-3.5 Turbo（OpenAI）
    • Zephyr（Hugging Face）
    • Qwen（通义千问，阿里）
    • Orca（Microsoft）
    • Claude Instant（Anthropic）

✅ 作用：在基础模型之上进行精调（fine-tuning），使其更适用于具体应用场景，比如客服、写作、代码生成等。

💡 可看作“定制化”的大模型，更贴近实际需求。

5.API Layer（API 层）

这一层提供了访问上述模型的接口服务，让开发者无需自己部署模型即可调用。

  • 常见API服务：
    • OpenAI API
    • Google PaLM / Gemini API
    • Microsoft Azure OpenAI
    • Cohere API
    • Anthropic API
    • Hugging Face Inference Endpoints
    • Stability AI API
    • Replicate API
    • A21 Labs API
    • ElevenLabs API

✅ 作用：简化开发流程，使企业或个人开发者能快速集成AI功能。

💡 比如你用ChatGPT写文章，其实就是在调用OpenAI的API。

6.Application Layer（应用层）

这是最上层，也是最终用户看到的产品和服务。

典型应用：
ChatGPT（OpenAI）

✅ 作用：将AI能力封装成易用的产品，服务于创作者、程序员、营销人员等。

💡 用户不需要懂AI，也能享受AI带来的便利。

✅ 总结：从下往上理解整个技术栈

层级 核心内容 类比
1. 基础设施 算力硬件（GPU/TPU/云） “房子的地基”
2. 架构 模型设计结构（Transformer等） “建筑图纸”

4. LLM 特定用途的优化模型 “装修后的房间”
5. API 接口服务 “水电管道”
6. 应用 最终产品（ChatGPT, Runway等） “家具和装饰”

编排模式

大语言模型（LLM）在实际应用中常用的六种“编排模式”（Orchestration Patterns）。这些模式不是直接关于模型本身的结构（比如Transformer），而是关于如何组织多个LLM调用、逻辑流程和外部系统交互，以完成更复杂任务。

这六种模式是构建高级AI应用的“设计模式”，类似于软件工程中的“设计模式”。每一种都解决一类特定问题：

  • 如何按顺序执行？
  • 如何并行处理？
  • 如何根据输入选择路径？
  • 如何自动优化结果？
  • 如何实现自动化代理？

🧩 1.Prompt Chaining（提示链）

描述：将多个LLM调用按顺序连接，前一个输出作为后一个的输入。如果失败则退出。

✅ 适用场景：

  • 多步骤推理（如先分析问题 → 再生成答案）
  • 错误处理机制（例如：第一步没成功就停止）

📌 例子：

  1. 用户提问：“帮我写一封辞职信。”
  2. 第一步：提取关键信息（职位、公司、原因）
  3. 第二步：基于信息生成草稿
  4. 第三步：润色语言风格

🔹 特点：串行、有条件跳转

🛣️ 2.Routing（路由）

描述：使用一个“路由器”决定下一步该调用哪个LLM，依据输入类型或条件。

✅ 适用场景：

  • 分类任务（如判断用户问题是客服还是技术问题）
  • 按照内容类型分发到不同模型（文本/代码/图像描述）

📌 例子：

  • 用户说：“我的电脑蓝屏了。” → 路由到“技术支持模型”
  • 用户说：“帮我写一首诗。” → 路由到“创意写作模型”

🔹 特点：动态决策、任务分流

⚡ 3.Parallelization（并行化）

描述：同时发起多个LLM调用，然后通过聚合器合并结果。

✅ 适用场景：

  • 多角度分析（比如从不同视角写摘要）
  • 多模型投票（提高准确性）
  • 同时生成多种风格的内容

📌 例子：

  • 让三个不同的模型分别生成同一新闻的不同版本摘要 → 然后融合成最终版
  • 用多个模型评估同一个代码片段的安全性

🔹 特点：并发执行、结果合并

👥 4.Orchestrator-workers（协调者-工作者模式）

描述：一个主控LLM（Orchestrator）分配任务给多个“工人”LLM，最后由合成器整合输出。

✅ 适用场景：

  • 复杂任务分解（如撰写报告）
  • 分工合作（如一人负责研究、一人写初稿、一人校对）

📌 例子：

  1. 主控模型：“我要写一篇关于气候变化的文章”
  2. 工人1：查找数据
  3. 工人2：总结观点
  4. 工人3：写引言
  5. 合成器：组合成完整文章

🔹 特点：任务分解 + 协同 + 整合

🔁 5.Evaluator-optimizer（评估-优化器）

描述：一个生成器提出方案，另一个评估器判断是否合格；如果不满意，返回修改直到达标。

✅ 适用场景：

  • 高质量内容生成（如法律文书、学术论文）
  • 自动化测试与迭代
  • 安全审核（防止偏见或错误）

📌 例子：

  1. 生成一段广告文案
  2. 评估是否符合品牌调性？有没有敏感词？
  3. 如果不行，给出反馈：“语气太强硬，请改得温和些”
  4. 重新生成 → 再次评估 → 直到接受

🔹 特点：闭环反馈、持续改进

🤖 6.Autonomous Agent（自主代理）

描述：LLM不断与环境互动，做出决策、采取行动，并学习反馈，形成闭环。

✅ 适用场景：

  • 自动化办公助手（自动查邮件、回复、安排会议）
  • 游戏AI角色
  • 机器人控制
  • Web自动化（如自动浏览网页、填写表单）

📌 例子：

  • 你告诉AI：“帮我订明天早上8点去机场的出租车。”
  • AI：
    1. 查询天气 → 发现下雨 → 建议提前出发
    2. 查找附近出租车平台 → 下单
    3. 收到确认 → 通知你
    4. 若失败 → 尝试备用方案

🔹 特点：持续交互、自主决策、记忆状态

🎯 总结对比表

模式 核心思想 适合任务 关键组件
Prompt Chaining 顺序执行，有分支 多步推理、流程控制 条件判断、失败退出
Routing 动态选路 分类处理、任务分发 路由器
Parallelization 并行执行 多视角分析、多模型投票 并发调用、聚合器
Orchestrator-workers 分工协作 复杂项目管理 协调者+工人+合成器
Evaluator-optimizer 迭代优化 高质量输出、安全审查 生成器+评估器+反馈
Autonomous Agent 自主循环 自动化、交互式任务 环境交互、状态记忆

💡 实际意义

这些模式告诉我们：

单个LLM的能力有限，真正的智能来自“编排”。

就像一个人可能很聪明，但一个团队才能完成复杂项目。LLM也是一样——通过合理的架构设计，可以让多个LLM协同工作，实现远超单一模型的功能。

🧠 思考延伸

  • 这些模式可以组合使用！比如：先用Routing分类 → 再用Parallelization生成多个版本 → 最后用Evaluator-optimizer筛选。
  • 当前主流框架（如LangChain、LlamaIndex、AutoGen）都在支持这些模式。
  • 未来趋势：从“调用LLM”到“训练LLM做代理”，即让LLM具备自我规划能力。

意图识别

意图识别（Intent Recognition）是自然语言处理（NLP）和人工智能（AI）领域的一项核心技术。它的核心目标是理解用户输入的文本或语音所表达的真实目的或意图。

简单来说，就是回答“用户想做什么？”这个问题。

概念

通俗易懂的例子

想象一下，您正在和一个智能助手对话：

  • 用户说：“帮我查一下明天上海的天气。”
    • 输入文本: “帮我查一下明天上海的天气。”
    • 意图识别结果: 查询天气 (Intent: Get_Weather)
  • 用户说：“我想听周杰伦的歌。”
    • 输入文本: “我想听周杰伦的歌。”
    • 意图识别结果: 播放音乐 (Intent: Play_Music)
  • 用户说：“订一张去北京的机票。”
    • 输入文本: “订一张去北京的机票。”
    • 意图识别结果: 预订机票 (Intent: Book_Flight)

核心作用

意图识别是构建各种智能对话系统（如聊天机器人、语音助手）的基础。一旦系统知道了用户的意图，它就能调用相应的功能模块来完成任务。例如：

  • 识别到 查询天气 意图后，系统会去调用天气 API 获取数据并返回。
  • 识别到 播放音乐 意图后，系统会启动音乐播放器。
  • 识别到 预订机票 意图后，系统会进入预订流程，询问出发地、时间等实体信息。

与“命名实体识别（NER）”的关系

意图识别通常和命名实体识别（Named Entity Recognition, NER）配合使用。

  • 意图识别：搞清楚用户“想干什么”。
  • 命名实体识别：从用户的话里“抠出”关键信息。

继续上面的例子：

  • 用户说：“帮我查一下明天（时间实体）上海（地点实体）的天气。”
  • 意图识别模块负责识别出这是一个 查询天气 的请求。
  • 命名实体识别模块则负责抽取出 时间：明天 和 地点：上海 这两个关键信息，以便查询天气的服务能够使用这些精确的参数。

总而言之，意图识别是让机器“听懂”人话的第一步，是实现智能化交互的关键环节。

意图识别的方法随着人工智能技术的发展，经历了从规则到机器学习再到深度学习的演进。目前主流的方法主要有以下几种：

意图识别有哪些方法

1.基于规则的方法 (Rule-Based)

这是最传统、最直观的方法。

  • 核心思想: 专家手工编写一系列的规则（if-else 语句、正则表达式等）来匹配特定的关键词、短语或句式，从而判断意图。
  • 优点:
    • 可控性强: 规则完全由人工制定，逻辑清晰，可以精确控制。
    • 准确性高: 对于领域明确、语料固定的场景，规则可以做到非常精准。
    • 无需训练数据: 不需要大量的标注语料。
  • 缺点:
    • 扩展性差: 需要大量人力维护和更新规则，非常耗时。
    • 泛化能力弱: 很难覆盖所有用户可能的表达方式，面对新问法容易失效。
    • 开发周期长: 对专家知识要求高。
  • 适用场景: 任务型对话系统、特定领域的客服问答、命令行解析等。

2.传统的机器学习方法 (Traditional Machine Learning)

这类方法将意图识别看作一个文本分类问题。

  • 核心思想: 首先对文本进行预处理（分词、去停用词），然后提取特征（如 TF-IDF、词袋模型 BOW），最后使用经典的分类算法（如朴素贝叶斯、SVM、逻辑回归）来训练模型。
  • 优点:
    • 性能稳定: 在标注数据充足的情况下，效果可靠。
    • 速度快: 训练和预测速度相对较快。
    • 可解释性较好: 可以通过特征权重来理解模型的决策依据。
  • 缺点:
    • 严重依赖特征工程: 特征提取的好坏直接影响模型效果，这需要很强的专业知识。
    • 泛化能力有限: 对于未见过的表达方式，识别能力不如深度学习模型。
  • 适用场景: 数据量中等，且对模型解释性有一定要求的场景。

3.深度学习方法 (Deep Learning)

这是当前最主流、效果最好的方法。

  • 核心思想: 利用神经网络（如 RNN、LSTM、GRU、CNN、BERT 等）自动从原始文本中学习特征和模式，进行端到端的意图分类。
  • 优点:
    • 泛化能力强: 能够很好地处理用户多变的表达方式，鲁棒性好。
    • 自动化特征提取: 无需人工设计特征，网络能自动学习到对任务有用的特征。
    • 效果卓越: 在大多数公开数据集上取得了 SOTA (State-of-the-Art) 的效果。
  • 缺点:
    • 需要大量标注数据: 模型性能高度依赖于高质量的训练数据。
    • 计算资源消耗大: 训练和部署可能需要较高的硬件配置。
    • 可解释性差: 模型内部决策过程像一个“黑盒”，难以解释。
  • 代表模型:
    • BERT及其变体: 如 RoBERTa, DistilBERT 等。这类模型通过预训练在大规模语料上学习通用的语言表示，然后在特定的意图识别任务上进行微调（fine-tuning），效果非常出色。
    • LSTM/GRU: 早期用于处理序列数据的强大模型，能捕捉上下文信息。

4.大语言模型方法 (Large Language Models, LLMs)

这是近年来随着 GPT、Claude 等大模型兴起而变得流行的新范式。

  • 核心思想: 利用大语言模型强大的理解和推理能力，通过提示工程（Prompt Engineering）或微调（Fine-tuning）来完成意图识别任务。
  • 优点:
    • 零样本/少样本学习能力: 在没有或只有很少训练数据的情况下，通过精心设计的提示词（prompt），大模型也能表现出不错的意图识别能力。
    • 上下文理解能力强: 能够结合对话历史等上下文信息进行更准确的判断。
    • 多任务处理: 一个模型可以同时处理意图识别、实体抽取等多种 NLP 任务。
  • 缺点:
    • 成本高昂: 调用 API 或部署大模型的成本较高。
    • 可控性差: 输出可能存在不可预测性，有时会产生幻觉。
    • 延迟较高: 相比传统模型，响应速度可能较慢。
  • 应用方式:
    • In-Context Learning (上下文学习): 将几个示例和待识别的句子一起作为输入，让模型模仿示例进行判断。
    • Fine-tuning: 在大模型基础上，使用少量自有数据进行微调，以适应特定业务场景。

基于规则的方法示例

基于规则的意图识别方法主要是通过人工定义的逻辑和模式匹配来判断用户输入的意图。虽然灵活性较差，但在特定领域和场景下依然非常有效和可靠。主要包括以下几种具体方法：

1.关键词匹配 (Keyword Matching)

这是最基础、最直接的方法。

  • 核心思想: 为每一个意图预先定义一个包含相关关键词的列表。当用户输入的文本中包含属于某个意图的关键词时，就判定为该意图。

  • 实现:

    • 创建一个字典，键是意图名称，值是关键词列表。
    • 将用户输入与每个意图的关键词列表进行匹配。
    • 返回匹配成功的意图。

  • 优缺点: 实现简单，但非常死板，无法处理同义词、近义词和复杂的句式变化。

2.正则表达式匹配 (Regular Expression Matching)

这是一种更强大的模式匹配方法，可以定义更复杂的文本模式。

  • 核心思想: 使用正则表达式（regex）来定义一个意图所匹配的文本模式。它可以匹配特定的词序、可选部分、重复模式等。

  • 实现:

    • 为每个意图编写一个或多个正则表达式。
    • 用这些表达式去匹配用户输入的文本。

  • 示例:

    • 这个 book_flight 的正则表达式可以匹配 “订一张去上海的机票”、”买个飞往北京的航班” 等多种说法。

  • 优缺点: 比纯关键词匹配灵活得多，能处理一定的句式变化，但编写复杂、全面的正则表达式非常困难，维护成本高。

3.语法树/依存句法分析 (Syntactic Parsing)

这种方法引入了语言学的语法知识。

  • 核心思想: 利用自然语言处理工具（如 Stanford CoreNLP, spaCy, NLTK）对句子进行句法分析，生成语法树或依存关系图。然后，通过分析句子的主谓宾结构、修饰关系等来判断意图。
  • 实现:
    • 预先定义每种意图在语法树上的典型结构模式。
    • 分析用户输入的句子，得到其语法结构。
    • 将该结构与预定义模式进行匹配。
  • 示例:
    • 意图 查询价格 可能对应一个模式：动词是 “查” 或 “看”，宾语是一个 “价格” 或包含 “多少” 的短语。
  • 优缺点: 能深入理解句子结构，对于复杂查询非常有效。但需要强大的 NLP 工具支持，对语法规则的定义也很复杂。

4.有限状态自动机 (Finite State Automaton, FSA)

这是一种形式化的方法，将意图识别过程看作在不同状态间跳转的过程。

  • 核心思想: 为每个意图设计一个有限状态机。状态机由状态和触发状态跳转的条件（如遇到特定词语）组成。用户输入的词语会驱动状态机从一个状态跳转到另一个状态，如果最终达到“接受状态”，则匹配成功。
  • 实现:
    • 为 预订酒店 意图设计一个自动机：开始 -> (遇到 “预订” 或 “开”) -> 等待地点 -> (遇到 “酒店” 或 “房间”) -> 等待时间 -> (遇到 “时间” 或数字) -> 结束(接受)。
  • 优缺点: 非常精确，适合处理有明确流程的意图。但对于意图多样、表达自由的场景，设计和维护大量状态机非常繁琐。

5.混合规则方法 (Hybrid Rule-Based Approaches)

在实际应用中，单一的规则方法往往效果不佳。因此，常常将上述多种方法结合起来使用。

  • 核心思想: 综合运用关键词、正则表达式、语法分析等多种规则，通过设定优先级或投票机制来得出最终的意图。
  • 示例:
    • 先用正则表达式处理高精度、有明显模式的意图。
    • 再用关键词匹配处理一些模糊的意图。
    • 最后根据匹配的强度或规则的优先级来决定最终结果。
  • 优缺点: 这是规则方法中最实用的一种，能在准确性和覆盖面之间取得较好的平衡，但系统会变得更加复杂。

什么是Agent？

Agent（智能体）是一个核心概念，它指的是一种能够感知环境、自主决策并采取行动以实现特定目标的软件实体。

把它想象成一个虚拟的、智能的“仆人”或“助手”。

Agent 的核心特征

一个典型的 AI Agent 必须具备以下几个关键特征：

  1. 自主性 (Autonomy): Agent 能够在没有人类或其他程序直接干预的情况下独立运行和决策。它不是被动地响应请求，而是可以主动地规划和执行任务。
  2. 反应性 (Reactivity): Agent 能够感知其所处环境的变化，并及时做出反应。例如，一个客服机器人能接收并回应用户的最新消息。
  3. 主动性 (Proactiveness): Agent 不仅仅是对环境做出反应，还能为了达成目标而主动采取行动。例如，一个股票交易 Agent 会根据市场趋势主动发出买入或卖出指令。
  4. 社交性 (Social Ability): Agent 能够与其他 Agent 或人类进行交互。例如，一个 Agent 可以调用另一个 Agent 或工具（如搜索引擎、计算器）来完成自己的任务。

Agent 的工作原理

通常，一个 Agent 的工作流程遵循 感知-思考-行动 (Sense-Think-Act) 的循环：

  1. 感知 (Sense): 通过传感器（在软件世界里，可能是 API、用户输入、数据库等）获取环境信息。
  2. 思考 (Think): 利用内置的逻辑、规则或更复杂的 AI 模型（如大语言模型）来分析感知到的信息，结合自身的知识和目标，进行推理、规划和决策。
  3. 行动 (Act): 根据决策结果，执行一个动作来影响环境。这个动作可以是输出一段文字、调用一个工具、修改数据库记录等。

Agent 的核心组成部分

核心组件

  1. Agent (智能代理): 图中用红色方框标出，是整个系统的“大脑”或“决策中心”。它负责接收输入、进行思考、做出计划，并决定下一步行动。
  2. Memory (记忆): 这是 Agent 的“记忆系统”，分为两种：
    • Short-term memory (短期记忆): 存储当前任务相关的、临时的信息，比如最近的对话内容、工具调用的结果等。
    • Long-term memory (长期记忆): 存储更持久的知识，比如用户偏好、历史交互记录、通用常识等。这些信息可以被 Agent 在不同任务间复用。
  3. Tools (工具): 这是 Agent 的“手脚”或“能力集合”。它是一组外部 API 或函数，Agent 可以调用它们来执行具体操作，例如：
    • Calendar(): 操作日历，查询或安排会议。
    • Calculator(): 执行数学计算。
    • CodeInterpreter(): 运行代码片段来解决编程问题。
    • Search(): 执行网络搜索获取实时信息。
    • ...more: 表示可以集成更多工具。
  4. Planning (规划): 这是 Agent 的“思考和策略制定”模块。它负责将一个大任务分解成一系列小的、可执行的步骤（子目标），并为每个步骤选择合适的工具。它还包含了多种高级推理方法：
    • Reflection (反思): 在执行完一步后，回顾结果是否符合预期，如果不符合，则调整后续计划。
    • Self-critics (自我批评): 对自己的推理过程进行批判性评估，检查是否有逻辑漏洞或错误。
    • Chain of thoughts (思维链): 一种推理方式，让 Agent 将其思考过程一步步地表达出来，形成一条清晰的逻辑链条。
    • Subgoal decomposition (子目标分解): 将复杂的总目标分解成多个更简单的子目标，然后逐一解决。
  5. Action (行动): 这是 Agent 的“执行单元”。根据 Planning 模块生成的计划，Action 模块会调用相应的 Tools 来执行具体的任务。

工作流程 (数据流)

图中的箭头表示了信息流动的方向：

  1. 输入: 用户的请求或任务会进入 Agent。

  2. 决策与规划: Agent 首先利用其 Memory 和 Planning 模块进行分析和思考。

    • 它会使用 Planning 模块来制定一个行动计划，可能包括使用哪些工具、按什么顺序使用。
    • 在这个过程中，它可能会调用 Reflection, Self-critics, Chain of thoughts, 或 Subgoal decomposition 等方法来辅助思考。

  3. 工具调用: 根据规划好的方案，Agent 通过 Action 模块去调用 Tools 中的某个具体功能（如 Search()）。

  4. 信息反馈: 工具执行后会产生结果（例如，搜索到的网页摘要）。这个结果会作为新的信息，通过 Action 模块反馈给 Agent。

  5. 记忆更新: Agent 会将这个新信息存储到 Memory 中，既可以是短期记忆（用于当前任务），也可以是长期记忆（用于未来参考）。

  6. 循环迭代: Agent 会持续地进行“思考-行动-观察-记忆”的循环，直到任务完成。

Agent 与传统程序的区别

  • 传统程序: 通常是线性的、基于预设规则的。输入一个指令，程序就执行一个固定的流程。它的行为完全由程序员编写的代码决定。
  • Agent: 是目标驱动的、适应性强的。它会根据当前环境和目标，自主选择最佳的行动路径。它的行为更具灵活性和智能性。

上下文理解与记忆管理

在 AI Agent 的体系结构中，记忆 (Memory) 是其能够进行连贯、智能和个性化交互的核心。它使 Agent 能够超越单次输入/输出的限制，具备了持续学习、适应环境和完成复杂任务的能力。

记忆的核心作用

  1. 维持上下文连贯性 (Contextual Continuity): Agent 需要记住对话或任务的历史，才能做出连贯的回应。例如，在多轮对话中，它需要知道用户之前说过什么、提过什么要求，以避免重复提问或前后矛盾。
  2. 积累知识与经验 (Knowledge & Experience Accumulation): Agent 可以将从交互、工具调用或外部知识库中获得的新信息存储起来，形成自己的知识库，从而随着时间推移变得越来越智能。
  3. 实现个性化服务 (Personalization): 通过记忆用户的偏好、习惯和历史行为，Agent 能够为不同用户提供定制化的服务和建议。
  4. 支持复杂任务执行 (Supporting Complex Tasks): 对于需要多步操作的任务（如计划、研究），记忆系统用于存储中间结果、当前进度和目标，确保任务能够顺利完成。
  5. 促进学习与反思 (Facilitating Learning & Reflection): Agent 可以回顾其决策过程和结果，进行自我评估和优化，这是实现 Agent 自我改进的关键。

记忆的分类

AI Agent 的记忆可以根据不同的维度进行划分，这种多维度的视角有助于构建更精细、更符合应用场景的记忆系统。

1.按存储时间分类

这是最基础也是最常用的划分方式。

  • 短期记忆 (Short-Term Memory):
    • 特点: 容量有限、时效性强、主要用于当前任务。
    • 作用: 存储与当前正在进行的交互或任务直接相关的临时信息。
    • 内容: 当前对话历史、任务的中间步骤和结果、临时变量。
    • 生命周期: 通常在单次会话或任务结束后被清除。
    • 技术实现: 通常由 Agent 框架管理，作为变量或状态传递给模型。
  • 长期记忆 (Long-Term Memory):
    • 特点: 容量巨大、持久存储、用于跨会话和跨任务的知识积累。
    • 作用: 存储持久性的、可复用的知识和信息。
    • 内容: 用户画像、历史交互记录、通用知识库、业务规则。
    • 生命周期: 持久化保存，跨越多个会话和任务。
    • 技术实现: 通常借助向量数据库（Vector Database）等外部存储，以支持高效的检索。

2.按存储内容分类

这种分类方式更深入地揭示了记忆内容的性质。

  • a. 情境记忆 (Episodic Memory):
    • 定义: 存储特定的交互事件或经历。这对应于 Agent 对每一次具体对话或任务的详细记录。
    • 在 Agent 中的应用: 记录与特定用户在特定时间点的完整对话流程、决策路径和结果。这有助于事后审计、复盘和在用户提及过往经历时进行追溯。
  • b. 语义记忆 (Semantic Memory):
    • 定义: 存储事实性知识和概念。这对应于 Agent 存储的通用知识和信息。
    • 在 Agent 中的应用: 存储世界的事实（如“巴黎是法国的首都”）、业务知识（如“公司报销政策”）、用户的具体信息（如“李四的生日是1990年”）。
  • c. 流程记忆 (Procedural Memory):
    • 定义: 存储如何执行任务的技能和程序。这对应于 Agent 的“技能库”或“行为策略”。
    • 在 Agent 中的应用: 存储 Agent 如何调用工具、执行特定任务的流程（如“处理退款请求的步骤”）。这可能是硬编码的逻辑，也可能是通过学习获得的策略。

3.按内容性质分类

  • 可描述（声明式, Declarative）: 指那些可以用明确的、结构化的方式表达出来的知识，如事实、规则。这与语义记忆高度重合。
  • 不可描述（非声明式, Non-declarative）: 指那些难以用语言明确表述，但会影响行为的知识或技能，如某种“直觉”或“做事的感觉”。这与流程记忆更为相关。

记忆的操作

AI Agent 的记忆管理系统需要执行一系列核心操作来维持其功能。

  1. 编码 (Encode):
    • 定义: 将原始数据（如用户输入的文本、工具返回的结果）转换成 Agent 内部可以理解和存储的表示形式。
    • 在 Agent 中的应用: 对于长期记忆，这通常意味着使用嵌入模型 (Embedding Model) 将文本转换为高维向量，以便后续存储和检索。
  2. 存储 (Storage):
    • 定义: 将编码后的信息保存到存储介质中。
    • 在 Agent 中的应用: 短期记忆存储在程序内存中；长期记忆通过 API 存储到向量数据库或其他数据库中。
  3. 提取 (Retrieval):
    • 定义: 在需要时，从存储中找回相关信息。
    • 在 Agent 中的应用: 当 Agent 需要外部知识时，它会将当前问题或上下文编码成向量，然后在向量数据库中进行相似度搜索，找到最相关的记忆片段，并将其注入到当前的上下文中。
  4. 巩固 (Consolidation):
    • 定义: 将不稳定或临时的记忆转化为更稳定、持久的记忆。
    • 在 Agent 中的应用: 指将短期记忆中的关键信息（如对话总结、用户偏好）提炼并存入长期记忆的过程。这确保了有价值的信息不会因会话结束而丢失。
  5. 反思 (Reflection):
    • 定义: 主动回顾、评估和检查记忆内容及决策过程。
    • 在 Agent 中的应用: 高级 Agent 会回顾自己的行动、工具调用结果和最终输出，评估其有效性，从中学习并优化未来的策略。反思的结论也可以作为新的记忆存储起来。
  6. 遗忘 (Forgetting):
    • 定义: 信息的自然流失或主动删除。
    • 在 Agent 中的应用: 这是一个必要的管理机制。短期记忆自然过期；长期记忆需要通过设置 TTL（生存时间）或使用遗忘算法来清理过时、冗余或不重要的信息，以保持系统的效率和相关性。

什么是 Function Calling？

Function Calling（函数调用） 是大语言模型（LLM）的一项关键能力，它允许模型在生成回答时主动调用外部工具或函数，从而获取实时数据、执行操作或访问私有信息，而不仅限于依赖其训练时的静态知识。

🎯 核心思想

让 LLM 从“只会说话”变成“能做事”的智能代理（Agent）

传统 LLM 只能基于训练数据“猜测”答案（例如：“巴黎天气可能是 15°C”），但通过 Function Calling，它可以：

  1. 判断是否需要外部信息
  2. 调用一个函数（如 get_weather("Paris")）
  3. 获取真实结果（如 {"temperature": 14}）
  4. 基于真实数据生成准确回答（“巴黎现在是 14°C”）

⚠️ 注意事项

  1. 不是模型直接执行代码
    LLM 只负责“决定调用什么函数+参数”，实际执行由你的系统完成（安全可控）。
  2. 需要结构化输出能力
    模型必须能生成符合 JSON Schema 的函数调用请求（现代 LLM 如 GPT-4、Claude 3、Llama 3 均支持）。
  3. 错误处理很重要
    函数可能失败（如网络错误、参数无效），需设计重试或降级策略。
  4. 避免过度调用
    可设置最大调用次数，防止无限循环（如 Agent 自问自答）。

💡 为什么重要？

  • 解决幻觉问题：用真实数据替代猜测
  • 扩展能力边界：连接现实世界（API、数据库、硬件）
  • 构建 AI Agent 基石：Function Calling 是实现自主代理（Autonomous Agent）的核心机制
