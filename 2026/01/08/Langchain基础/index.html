LangChain基础

LangChain是开始构建由大型语言模型（LLMs）驱动的智能体和应用程序的最简单方法。只需不到10行代码，就可以连接到OpenAI、Anthropic、谷歌以及更多平台。LangChain提供了预先构建的智能体架构和模型集成，帮助你快速上手，并将大型语言模型无缝整合到你的智能体和应用程序中。

核心组件

Models

在 LangChain 中，**Models 模块（或更准确地说，模型相关组件）的核心作用是：为应用程序提供统一、标准化的接口，用于调用各种大语言模型（LLM）和聊天模型（Chat Model），无论其来自 OpenAI、Anthropic、Google、开源模型（如 Llama）还是本地部署。**

模型可以通过两种方式使用：

  1. 使用智能体时：创建[智能体]时可以动态指定模型。
  2. 独立式：模型可以直接调用（在智能体循环之外），用于文本生成、分类或提取等任务，无需智能体框架。
初始化模型

在LangChain中开始使用独立模型最简单的方法是使用init_chat_model从你选择的聊天模型提供商那里初始化一个（示例如下）：

uv add "langchain[openai]"

调用方式

必须调用聊天模型才能生成输出。主要有三种调用方法，每种方法适用于不同的使用场景。

invoke

调用模型最直接的方法是使用invoke()，传入单条消息或消息列表。

可以向聊天模型提供消息列表来表示对话历史。每条消息都有一个角色，模型用该角色来指示对话中是谁发送了这条消息。

Stream流

大多数模型在生成输出内容时都能进行流式传输。通过逐步显示输出，流式传输显著改善了用户体验，尤其是对于较长的响应而言。

调用stream()会返回一个iterator 迭代器,它会在生成输出块时将其产出。你可以使用循环来实时处理每个块。

与invoke()不同，后者会在模型完成其完整响应生成后返回单个AIMessage，而stream()会返回多个AIMessageChunk对象，每个对象都包含输出文本的一部分。重要的是，流中的每个块都旨在通过汇总组合成一条完整的消息。

生成的消息可以与使用invoke()生成的消息同等对待——例如，它可以被聚合到消息历史中，并作为对话上下文传递回模型。

Batch批量

默认情况下，batch() 只会返回整个批次的最终输出。如果你希望在每个单独的输入完成生成时就收到其输出，可以使用 batch_as_completed() 来流式传输结果。

聊天模型方法batch()，该方法在客户端并行处理模型调用。与推理提供商（如OpenAI或Anthropic）支持的批量API不同。

使用 [batch_as_completed()] 时，结果可能会乱序到达。每个结果都包含输入索引，以便在需要时通过匹配来重建原始顺序。

Messages

在 LangChain 中，消息是模型上下文的基本单位。它们代表模型的输入和输出，承载着与大语言模型（LLM）交互时表示对话状态所需的内容和元数据。

消息是包含以下内容的对象：

  • 角色 —— 标识消息类型（例如系统、用户）
  • 内容 —— 代表消息的实际内容（如文本、图像、音频、文档等）
  • 元数据 —— 可选字段，如响应信息、消息 ID 和令牌使用情况

LangChain 提供了一种标准的消息类型，适用于所有模型提供商，确保无论调用何种模型，都能有一致的行为。

基本用法
Text prompts

适用场景：

  • 单一的、独立的请求
  • 不需要对话历史
  • 希望代码复杂度最低
Message prompts

可以通过提供消息对象列表，向模型传入消息列表。

适用场景：

  • 管理多轮对话
  • 处理多模态内容（图像、音频、文件）
  • 系统指令
Dictionary format

可以直接以 OpenAI 聊天补全格式指定消息。

消息类型
System message

SystemMessage代表一组初始指令，用于引导模型的行为。你可以使用系统消息来设定语气、定义模型的角色，并建立响应准则。

Human message

一个HumanMessage代表用户输入和交互。它们可以包含文本、图像、音频、文件以及其他任何形式的多模态内容。

消息元数据

AI message

AIMessage 表示模型调用的输出。它们可以包含多模态数据、工具调用以及你之后可以访问的特定于提供商的元数据。

AIMessage对象在调用模型时由模型返回，其中包含响应中的所有相关元数据。

提供商对不同类型的消息会进行不同的权衡和情境分析，这意味着有时手动创建一个新的AIMessage对象并将其插入消息历史中，假装它来自模型，会很有帮助。

消息内容

你可以将消息的内容视为发送给模型的数据负载。消息具有一个 content 属性，该属性为松散类型，支持字符串和非类型化对象的列表（例如字典）。这使得 LangChain 聊天模型可以直接支持提供商原生结构，例如多模态内容和其他数据。

此外，LangChain 为文本、推理、引用、多模态数据、服务器端工具调用和其他消息内容提供了专用的内容类型。

LangChain 聊天模型接受 content 属性中的消息内容,可能包含以下任一内容

  • 一个字符串
  • 以提供商原生格式呈现的内容块列表
  • LangChain的标准内容块列表

标准内容块
LangChain 提供了一种适用于所有提供商的消息内容标准表示形式。

消息对象实现了一个content_blocks属性，该属性会将content属性延迟解析为标准的、类型安全的表示形式。例如，从ChatAnthropic或ChatOpenAI生成的消息会包含各自提供商格式的thinking（思考）或reasoning（推理）块，但可以被延迟解析为一致的ReasoningContentBlock表示形式。

多模态

多模态指的是处理不同形式数据的能力，例如文本、音频、图像和视频。LangChain包含了可跨提供商使用的这些数据的标准类型。

聊天模型可以接受多模态数据作为输入并生成多模态数据作为输出。下面我们展示几个包含多模态数据的输入消息的简短示例。

跨提供商标准类型的含义：

  1. 统一的消息格式
    LangChain 定义了标准的 HumanMessage、AIMessage 等消息类型，
    无论底层是 OpenAI、Anthropic 还是 Google，都使用相同的接口
  2. 标准化的多模态数据结构
    图像：统一使用 {“type”: “image_url”, …} 格式
    文本：统一使用 {“type”: “text”, …} 格式
    LangChain 在内部转换为各供应商的特定格式
  3. 供应商无关的代码
    更换 AI 提供商时无需修改消息构建代码
    只需更改模型初始化部分
    降低了供应商锁定风险
  4. 抽象化处理
    LangChain 屏蔽了各供应商 API 的差异
    开发者只需关注业务逻辑，不需关心底层实现细节
    便于 A/B 测试不同供应商的模型性能
    这就是所谓的”跨提供商标准类型”——为不同的 AI 服务提供了统一的接口和数据格式。

工具

工具扩展了智能体的能力——使它们能够获取实时数据、执行代码、查询外部数据库以及在现实世界中采取行动。

从本质上来说，工具是具有明确定义的输入和输出的可调用函数，这些函数会被传递给聊天模型。该模型会根据对话上下文决定何时调用工具，以及提供哪些输入参数。

创建工具

创建工具最简单的方法是使用@tool装饰器。默认情况下，函数的文档字符串会成为工具的描述，帮助模型了解何时使用该工具。

类型提示是必需的，因为它们定义了工具的输入模式。文档字符串应内容丰富且简洁，以帮助模型理解工具的用途。

访问上下文

重要性何在：当工具能够访问智能体状态、运行时上下文和长期记忆时，它们的功能最为强大。这使工具能够做出具有上下文感知的决策、提供个性化响应，并在对话中保留信息。

运行时上下文提供了一种在运行时将依赖项（如数据库连接、用户ID或配置）注入工具的方法，使这些工具更易于测试和复用。

工具可以通过ToolRuntime参数访问运行时信息，该参数提供：

  • State - 在执行过程中流动的可变数据（例如，消息、计数器、自定义字段）
  • Context - 不可变配置，如用户ID、会话详情或特定于应用程序的配置
  • Store - 跨对话的持久性长期记忆
  • Stream Writer - 在工具执行时流式传输自定义更新
  • Config - 用于执行的RunnableConfig
  • Tool Call ID - 当前工具调用的ID

使用ToolRuntime在单个参数中访问所有运行时信息。只需将[runtime: ToolRuntime]添加到工具签名中，它将被自动注入，且不会暴露给大语言模型。

短期记忆

核心概念总结

1. 什么是短期记忆：

LangChain 中的短期记忆，是指智能体在单个对话线程（thread） 内，对历史交互信息的临时存储与调用能力，核心是保留单轮对话的上下文，让 AI 能够连贯理解用户的连续提问。

  • 线程（thread）：相当于一次独立的对话会话，类似邮件的单条对话分组，不同线程的记忆完全隔离，互不干扰。
  • 核心载体：以对话消息列表（messages）为主要形式，记录用户输入与 AI 回复的完整历史。
  • 核心问题：大模型存在上下文窗口限制，过长的对话历史会超出窗口导致信息丢失、模型效果下降、响应变慢且成本增加。

2. 短期记忆的核心价值：

  • 实现对话连贯性，让 AI 记住用户在当前对话中提及的信息（如姓名、偏好、历史问题）。
  • 支持智能体在多步骤任务中，复用之前的交互结果与工具调用信息。
  • 通过持久化存储，支持对话线程的中断与恢复。

3. 核心工作机制：

LangChain 通过状态（State）+ 检查点（Checkpointer） 实现短期记忆：

  1. 智能体的对话历史、自定义信息都存储在AgentState中，作为记忆的核心载体。
  2. Checkpointer 负责将状态持久化到内存或数据库，实现线程隔离与记忆保存。
  3. 每次调用智能体或完成工具调用时，记忆自动更新；每次执行步骤前，自动读取历史状态。
基础使用
内存版短期记忆（开发调试用）

使用 InMemorySaver 将记忆存储在程序内存中，适合本地开发、测试场景，重启程序后记忆丢失。

代码说明:

  • thread_id：核心参数，用于区分不同的对话线程，相同thread_id共享同一份记忆。
  • InMemorySaver：内存级检查点，无需数据库，开箱即用。
  • 多次调用agent.invoke，传入相同config，即可实现连续对话的记忆保留。
生产版短期记忆

生产环境中，内存存储无法满足需求，需使用数据库持久化记忆，支持多实例共享、记忆永久保存。

自定义记忆状态

默认的 AgentState 仅包含 messages 字段存储对话历史，实际业务中需要存储用户 ID、偏好等自定义信息，可通过扩展 AgentState 实现。

核心说明:

  • 自定义状态需继承AgentState，基于 Pydantic 定义字段类型与默认值。
  • 调用agent.invoke时，可直接传入自定义字段，智能体会自动将其纳入短期记忆。
  • 自定义状态会和对话历史一起，通过 Checkpointer 持久化存储。
常见模式

启用 短期记忆 后，长时间对话可能会超出大语言模型（LLM）的上下文窗口。

针对长对话超出大模型上下文窗口的问题，LangChain 提供四种主流处理方案，以下是每种方案的可执行代码与使用场景。

裁剪消息

决定何时截断消息的一种方法是计算消息历史中的令牌数量，并在接近该限制时进行截断。如果使用LangChain，可以使用修剪消息工具，并指定要从列表中保留的令牌数量，以及用于处理边界的strategy（例如，保留最后的max_tokens）。

保留最新的部分消息，保留系统提示词 + 最近交互记录，是最轻量化的解决方案，通过@before_model中间件实现。

删除消息

主动删除指定或全部历史消息，适用于需要清理敏感信息、重置对话上下文的场景，通过RemoveMessage实现。

总结消息

将早期的长对话历史总结为精简文本，替换原有历史，既保留关键信息，又减少 token 占用，是长对话的最优解。

记忆的读写与操作
在工具中读取短期记忆

通过ToolRuntime参数，工具可直接读取智能体的状态（包括对话历史、自定义字段）。

在工具中写入短期记忆

通过Command，工具可直接修改智能体的状态，实现记忆的动态更新。

动态提示词中使用记忆

通过@dynamic_prompt中间件，读取记忆中的信息，生成动态的系统提示词。

流式输出

核心概念总结

1. 什么是 LangChain 流式输出：

LangChain 提供完善的流式系统，可在智能体（Agent）运行过程中，实时推送状态更新、LLM 生成的词元、自定义信息，无需等待完整响应生成完毕，大幅提升应用的响应速度和用户体验，解决大模型推理延迟的问题。

2. 核心流式模式：

LangChain 支持三种基础流式模式，可单独使用或组合使用，所有模式通过 stream/astream 方法指定：

模式 核心作用 输出内容
updates 流式输出智能体执行进度 每个步骤后的状态更新，如 LLM 节点的工具调用、工具节点的执行结果
messages 流式输出 LLM 生成词元 (token, metadata) 元组，包含逐词生成的内容和节点元信息
custom 流式输出自定义信息 通过流写入器，在工具 / 节点中推送自定义文本、数据等
