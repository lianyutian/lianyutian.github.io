ReentrantLock详解

1. 锁示例

基于JDK17

今天来看下ReentrantLock锁实现的原理，不多BB先看一段代码。

这是一段很简单的代码，定义个售票中心。售票中心里定义了共享变量capacity车票数。再定义一个售票窗口类，负责车票的售卖。很明显此时会有多线程并发问题，多个窗口同时卖票时sale()方法未加锁，多个线程会同时操作capacity变量。此时会导致超卖的现象。所以我们要对sale方法加把锁。

加锁后同一时间只能有一个线程来访问下面的售票逻辑，不会导致超卖现象。接下来就让我们深入lock()方法，探究下lock()的加锁逻辑。

2. 加锁流程

来看下这个方法的定义：

  1. 申请一把锁
  2. 如果锁没有被其他线程持有，那么立马返回并锁定也就是设置state=1
  3. 如果锁已经被当前线程持有了，那么state+1然后立马返回
  4. 如果锁被其他线程持有，那么当前线程就不可用（阻塞）直到申请到锁。同时也会设置state=1

再来看下实现：

initialTryLock()该方法有两个实现，公平锁和非公平锁：

我们先看非公平锁：

  1. 首先获取当前线程
  2. 直接尝试使用CAS去加锁（这也是为什么称为非公平的原因）
    • 加锁成功，设置持有这把锁的线程为当前线程
    • 返回true加锁成功
  3. else if (getExclusiveOwnerThread() == current)这一行判断就是重入逻辑了，如果持有这把锁的就是当前线程那么将state值+1然后返回true加锁成功。
  4. 剩余就是加锁失败了。

好，现在根据返回值再返回看看：

  1. 加锁成功返回true，没啥可说的。
  2. 加锁失败时，就去执行acquire(1)方法。

现在我们总结下整个加锁流程：

  1. 调用lock()方法尝试加锁；
  2. 加锁不成功调用acquire(1)；
  3. 调用tryAcquire(arg)再尝试获取锁；
  4. 获取锁失败调用acquire(null, arg, false, false, false, 0L);
  5. 进行5轮for循环；
  6. 第一轮for循环：将node赋值为一个独占的ExclusiveNode；
  7. 第二轮for循环：node.waiter设置为当前线程，赋值h为一个独占节点，将对象的HEAD字段的值从null更新为h，此时HEAD = TAIL = h；
  8. 第三轮for循环：走完循环，此时链表为HEAD -> node = TAIL。
  9. 第四轮for循环：将node的状态设置为等待标识。
  10. 第五轮for循环：将当前线程park。

4. 公平锁加锁

主要就是多了这么一个判断，通过从后向前遍历判断链表中是否有正在等待的线程节点。如果有就不去CAS抢锁，没有等待节点就尝试去获取锁。

5. 解锁流程

  1. 释放一把锁
  2. 如果是持有锁的当前线程那么将state - 1，当state值为0的时候释放锁。
  3. 如果不是持有锁的线程进入该方法那么抛出异常

唤醒后的线程会从lock中LockSupport.park(this)处开始执行，再次开始for循环

ReentrantLock的加解锁流程到这就说的差不多了，ReentrantLock使用的是AQS中的独占模型。共享模型可以看下CountDownLatch。
