1.1 跨域问题

1.1.1 概念

跨域问题是指在浏览器中执行一个网页向另一个不同源的服务器发送请求时，由于同源策略（Same-Origin Policy，SOP）的限制，浏览器会阻止这个请求。同源策略是一种用于保护用户信息安全的浏览器安全机制，它限制了来自不同源的资源互相访问。

1.1.2 什么是“同源”？

在浏览器中，同源指的是以下三个部分都相同：

  1. 协议（如 http:// 和 https://）
  2. 域名（如 example.com）
  3. 端口（如 80 或 443）

只有当这三个部分完全一致时，两个URL才被认为是同源的。否则就是跨域。例如：

  • http://example.com 和 http://example.com:8080 不是同源，因为端口不同。
  • http://example.com 和 https://example.com 不是同源，因为协议不同。
  • http://example.com 和 http://anotherdomain.com 不是同源，因为域名不同。

1.1.3 常见的跨域场景

  1. 前端调用后端API：前端页面在 http://example.com 下运行，而后端API服务器可能在 http://api.example.com 上，属于不同源。
  2. CDN资源：网页可能会从不同的源（如CDN）加载JavaScript、CSS或图片资源。
  3. 跨域请求（CORS）：通过JavaScript（如fetch或XMLHttpRequest）向不同源的服务器发送请求。

1.1.4 如何解决跨域问题？

  1. CORS（跨域资源共享，Cross-Origin Resource Sharing）：最常见的解决方案。通过设置服务器的HTTP响应头中的 Access-Control-Allow-Origin，服务器可以指定允许哪些源的请求。常见的CORS头包括：
    • Access-Control-Allow-Origin: 指定允许的源，或者使用 * 表示允许所有源。
    • Access-Control-Allow-Methods: 允许的HTTP方法（如 GET、POST 等）。
    • Access-Control-Allow-Headers: 允许的请求头。
  2. 反向代理：通过设置反向代理服务器（如 Nginx、Apache）将请求代理到目标服务器，从而让请求看起来是从同一个源发出的。

1.3 密码加密

1.3.1 简介

bcrypt 是一种用于安全存储密码的加密哈希算法。bcrypt 常用于加密密码，确保即使密码被泄露，原始密码也难以被恢复。

1.3.2 工作原理

  1. 盐（Salt）：每次加密时，bcrypt 会为密码生成一个随机的“盐值”。这个盐值用于加强密码的安全性，即使两个用户使用相同的密码，加密后的结果也会不同。
  2. 哈希（Hashing）：密码与盐值结合后通过 bcrypt 算法进行多次迭代哈希，生成一个加密后的密码。
  3. 验证：在验证密码时，bcrypt 会将输入的密码与加密后保存的哈希值进行比较，看它们是否匹配。

1.3.3 基础使用

使用 bcrypt 的步骤：

  1. 安装 bcrypt 包： 你可以使用 Go 的官方包管理工具 go get 来获取该包：

  3. 验证密码： 使用 CompareHashAndPassword 函数比较用户输入的密码与数据库中保存的 bcrypt 哈希值。

1.4 登录校验

1.4.1 Cookie和Session

HTTP 是一种 无状态 协议，意味着每个请求都是独立的，与之前或之后的请求没有直接关联。服务器不会自动保存客户端之间的会话信息。

如何管理状态：尽管 HTTP 是无状态的，但可以通过以下机制来模拟有状态的会话：

  • Cookies：服务器通过设置 cookie 将状态保存在客户端浏览器中，随后浏览器每次请求时都会自动发送这些 cookie 信息，维持会话状态。（也正因为 Cookie 是放在浏览器本地的，所以很不安全）

1.4.4 Session 刷新

1.4.4.1 场景

这里有一个登录状态很经常遇到的问题，就是我们这 Session id 所在的 Cookie，过期时间是固定的。

举个例子：假如你设置为 10 分钟，那么用户登录了 9:59 秒之后，还能访问网站，结果过了两秒，他就被要求重新登录。

也就是需要在用户持续使用网站的时候，刷新过期时间。

1.4.4.2 解决方法

如何刷新？

  • 用户每次访问，我都刷新。

    性能差，对 Redis 之类的影响很大。

  • 快要过期了我再刷新，比如说 10 分钟过期。当用户第 9 分钟访问过来的时候，我就刷新。

    万一我在第 9 分钟以后都没再访问过呢？

  • 固定间隔时间刷新，比如说每分钟内第一次访问我都刷新。

  • 使用长短 token。这个我们在后面接入微信登录的时候再深入讨论。

1.4.5 JWT

JWT（JSON Web Token）是一种开放标准（RFC 7519），用于在各方之间以 JSON 对象安全地传输信息。JWT 通常用于身份验证和授权机制，它的设计轻量且跨平台，可以用于 Web、移动设备或其他应用程序。

1.4.5.1 JWT 的基本结构

JWT 包含三个部分：

  1. Header（头部）：说明签名算法等信息。
  2. Payload（负载）：包含实际传输的数据，通常是用户信息或其他声明。
  3. Signature（签名）：用于验证发送方的身份并确保消息未被篡改。

JWT 的格式为：Header.Payload.Signature，三部分之间用点 . 分隔。

1. Header（头部）

Header 通常由两部分组成：

  • **alg**：签名算法，如 HMAC SHA256 或 RSA。
  • **typ**：表示 token 的类型，通常是 “JWT”。

2. Payload（负载）

Payload 是 JWT 的主体部分，包含声明（claims）。声明可以是预定义的，也可以是自定义的。

一些常见的标准声明：

  • **iss**（Issuer）：签发者
  • **sub**（Subject）：主题
  • **exp**（Expiration Time）：过期时间
  • **aud**（Audience）：接收者
  • **iat**（Issued At）：签发时间

1.4.5.2 JWT 的生成与验证过程

1. 生成 JWT

  • 服务器在用户通过认证（如登录）后，生成一个 JWT，并将其返回给客户端。
  • 客户端收到 JWT 后，通常会将其存储在 localStorage 或 HTTP Cookies 中。

2. 客户端发送 JWT

  • 每次客户端请求受保护的资源时，都会将 JWT 放在 HTTP 请求的 Authorization 头中：

3. 验证 JWT

  • 服务器接收到请求时，会从请求头中提取 JWT，并通过签名验证 JWT 的合法性和完整性。
  • 验证通过后，服务器将根据 JWT 的 payload 信息进行相应的处理。

1.4.5.3 优缺点

优点

  • 无状态：服务器不需要存储会话数据，JWT 完全存储在客户端，减少服务器压力。
  • 跨平台：JWT 是 JSON 格式的，支持多种语言和平台。
  • 灵活：可以在 Payload 中自定义存储用户信息或其他声明。

缺点

  • 安全性问题：如果 JWT 未加密且泄露，敏感信息可能被暴露。
  • 不可撤销：JWT 是无状态的，生成后无法在服务器端主动使其失效（除非设计特殊的黑名单机制）。
  • 大小较大：与传统的 Session ID 相比，JWT 更大，因为包含了完整的信息。

1.7.1 作用
